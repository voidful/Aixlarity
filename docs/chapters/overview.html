架構

Aixlarity 的完整架構一覽

Aixlarity 的整個系統長什麼樣？沿著 CLI → App → 提示詞 → AI 對話 → 工具 → 記錄 這條路，看看每個零件接在哪裡。

怎麼判斷功能是不是真的存在

只要某個功能沒有出現在真正的執行流程（App::exec 或 run_agent）裡，我們就不把它算成「已完成」。這一章的所有判斷，都是根據實際程式碼核對後得出的。

剛看完入門章？

只需要帶著一個問題往下看：剛才那個小範例裡的四個步驟，在 Aixlarity 裡實際上是由哪些程式碼負責的。先不用記每個檔名，只要看懂「輸入去哪、提示詞在哪裡組合、工具在哪裡跑、結果在哪裡存」就好。

現在的架構要同時看 CLI 和 IDE

Aixlarity 的 harness runtime 在 Rust core，但最容易教學的 control surface 在 aixlarity-ide。CLI 展示「一個任務如何跑完」；IDE 展示「任務如何被批准、審核、回放、比較、附證據」。所以讀架構時，請把 aixlarityView.ts 視為產品層入口，把 crates/aixlarity-core 視為 runtime 層入口。

先沿著「主要流程」看全貌

很多專案的文件會從模組清單開始介紹，但這容易把「程式碼存在」和「程式真的會執行」搞混。更好的方式是從一條真正跑得動的流程出發，順著輸入一路看到提示詞、工具、記錄和輸出。

你應該先問的問題 在 Aixlarity 對應哪一層 為什麼重要
使用者是在 CLI 還是 IDE 操作 aixlarity-cli 或 aixlarity-ide IDE 會多帶 editor context、approval UI、artifact review 和 durable Mission Control。
使用者輸入先去哪裡 aixlarity-cli 與 App::handle 這決定 CLI 只是外殼，還是偷偷塞滿產品邏輯。
工作區、信任與 provider 何時載入 App::exec 這會決定 prompt 前的準備工作是否清楚、可測、可追蹤。
Prompt 到底怎麼被組裝 prompt.rs 這一層最能看出 agent 是不是只靠使用者當場描述任務。
Tool call 誰來執行與限制 agent.rs 與 tools/ 這直接關係到操作影響與安全設計是否真的存在。
結果是否會被保存與回用 session/ 沒有這一層，就談不上 replay、resume、fork 與持續工作流。

一條真正跑得動的主要流程

來源：crates/aixlarity-cli/src/main.rs

CLI 與 REPL 入口

CLI 參數、REPL、JSON/JSONL 輸出與命令映射都已接好。這一層的作用是把使用者任務轉成 AppCommand，不是在 CLI 端自行堆太多產品邏輯。

來源：crates/aixlarity-core/src/app.rs

Workspace / Trust / Provider / Catalog 載入

App::exec 會依序做工作區探測、trust 評估、provider 載入、commands / skills / instructions / plugins 載入，再進 prompt 組裝。這代表 Aixlarity 不是單純把輸入直接丟給模型，而是先完成一次 外殼 組態展開。

來源：crates/aixlarity-core/src/prompt.rs、prompt/expansion.rs

提示詞組裝

指令檔、命令模板、技能、附件注入與需要審核的指令都會被組成最終 prompt。這是 Aixlarity 最接近 Claude Code / Codex 風格的地方，因為它把倉庫語境與任務語境分成明確層次。

來源：crates/aixlarity-core/src/agent.rs、agent/adapters/

Provider Adapter 與 代理迴圈

Gemini、OpenAI Responses / Chat、Anthropic Messages 都有實際 adapter；工具迴圈、fallback、streaming、token 用量、context compacting 也都已進主要流程。這一層屬於真正會動的 runtime，不只是介面設計。

來源：crates/aixlarity-core/src/tools/、plugins.rs、mcp.rs

工具、插件、MCP、Coordinator

內建工具、plugin tools、MCP tool loading、sub-agent coordinator 都會被合併進工具註冊表。內建工具完整覆蓋讀寫搜尋執行；plugins 已接入 live tool registry；MCP client 支援 stdio 連線、工具探測與執行；container sandbox 提供 Docker / Podman 執行隔離。

來源：crates/aixlarity-core/src/session/

Session / Checkpoint / Replay

checkpoint、session 儲存、resume / fork / replay 與事件紀錄都有實際資料結構和測試。這一層讓 Aixlarity 不必把每次執行都當成孤立 prompt。

從 App::exec 看一次主要流程展開

如果只看全景卡片，還是有點抽象。真正的關鍵，在於 App::exec 把哪些事情明確放在 live 執行時 之前。

步驟 1：先探測工作區，再決定 trust

Workspace::discover 會往上找 .aixlarity、.git、Cargo.toml、package.json。接著 TrustStore::evaluate 判定這個工作區是 trusted、trusted-via-parent 還是 untrusted。這一步的重要性在於，repo instructions 與 workspace config 能不能進系統，不是單靠檔案存在與否，而是先過 信任檢查。

步驟 2：把 providers、commands、skills、instructions、plugins 都載進 runtime

這一段很值得注意，因為它顯示 Aixlarity 把 command loading、skill loading、提示詞組裝 視為不同層。換句話說，agent 的「工作方式」不是靠一份超長系統 prompt 暗藏，而是先分層讀入，再由 prompt 組裝階段決定如何併合。

步驟 3：組 prompt，必要時插入 工作接續

assemble_prompt 先處理 command invocation、skill 選用、檔案注入與 shell approval 區塊；如果是 resume / fork，則會把 render_session_context 產生的 continuation 摘要插到 task 前面。這代表 session 並不是旁邊另存一份記錄而已，而是會真實回傳到下一輪工作。

步驟 4：先決定是不是 plan-only，再決定是否進 live 代理迴圈

如果使用者選的是 plan_only，Aixlarity 會保存 checkpoint / session 相關事件，但不會真正進到 run_agent。這點很重要，因為它代表目前的 規劃模式 比較像「規劃輸出模式」，而不是一條完全獨立的互動 runtime。

步驟 5：進 live 代理迴圈，最後回寫 session 與 execution events

真正的工具執行、provider 呼叫、fallback、hooks、工具記錄 與 token 用量 都發生在 run_agent。完成後再經過 save_new_session 或 append_session_turn 寫回 session。換句話說，工作記錄保存 是 執行時 尾端的正式產物，不是另寫一份「紀錄用檔案」而已。

範例：同一個任務在 trusted 與 untrusted 工作區的差別 
Task: /review src/app.rs

Case A: trusted workspace
- 會載入 AGENTS.md / AIXLARITY.md
- 會讀取 workspace commands / skills / providers
- 最終 prompt 帶有完整 repo workflow

Case B: untrusted workspace
- instructions bundle 會變成空
- workspace-scoped commands / skills / providers 會被限制
- agent 仍可工作，但會失去很多 repo 專屬語境

重點：
這不是 UX 小差異，而是 信任檢查 對 提示詞組裝 的直接影響。

三個完成度判讀範例

光看矩陣容易記住結論，卻不知道判斷是怎麼來的。下面挑三個最容易誤判的項目，示範 實況檢查 的推理方式。

項目 常見誤讀 實際判讀方式 結論
Container sandbox 有 container.rs 就代表所有工具都在容器裡跑。 回到 tools/shell.rs 看，會發現目前只有 shell 在 sandbox=container 時改走容器路徑。 Partially Wired：是 shell sandbox，不是全工具隔離。
Memory / MEMORY.md 有 memory 讀寫函式，就代表系統已有長期記憶。 回頭追 App::exec、assemble_prompt、run_agent，目前看不到 memory 穩定回傳到主要流程。 Partially Wired：有持久化介面，仍需主要流程驗證。
Planning mode 有 planning 參數，就代表存在一條獨立的規劃 agent runtime。 App::exec 的 plan_only 會保存相關事件與輸出，但不會進 live run_agent 工具迴圈。 Implemented as output mode：這是規劃輸出模式，不是完整獨立 planning runtime。

哪些能力名字很大，但現在要保守讀

M

MEMORY.md

有讀寫函式，但目前沒有在 提示詞組裝 或 工作接續 主要流程中持續回傳。它比較像已經準備好一段持久化介面，而不是成熟的 long-term memory 系統。

C

Container Sandbox

container.rs 能把 shell 指令送進 Docker / Podman，但其他讀寫工具仍直接作用在工作區。這表示「sandbox=container」目前還不是完整的全工具隔離含義。

P

MCP

Aixlarity 會從 .aixlarity/mcp.json 載入 MCP 伺服器，完成初始化與 tools/list 探測，再把工具包進 runtime；但 client 實作仍偏簡化，相容性與錯誤處理描述都該保守。

T

Token Caching

目前原始碼還看不到穩定 快取金鑰、請求簽名 或 provider-specific cache path。文件可以把它當成參考方向，但不能寫成既有主要流程能力。

把一個任務放進全景裡看

用一個具體例子會更容易理解這個全景。假設使用者執行 aixlarity exec "/review src/app.rs"，實際上會依序發生這些事：

CLI 層把輸入變成 AppCommand::Exec

CLI 的責任不是自己判斷 prompt 細節，而是把輸入、sandbox、provider、permission、session 選項整理成一個清楚的執行請求。

App facade 決定「這次工作」的邊界條件

工作區根目錄、trust 狀態、目前 provider、可用 commands / skills / plugins，都是在這裡被確定。這也是為什麼我們說 App::exec 是 外殼 的總閘門。

Prompt assembly 把 workflow 與 repo 語境寫進最終 prompt

如果 /review 是自訂命令，模板會展開；如果輸入包含 @{file}，文字檔會注入預覽；如果有 shell 區塊，則會被轉成待批准區段。這不是單純的字串拼在一起，而是把不同來源的控制資訊變成一份 可送出的提示詞。

Agent loop 讓模型真的能工作

接著模型可以要求讀檔、搜尋、跑 shell、套 patch。每一次工具呼叫都會先過 permission 與 hooks，再落到具體工具實作。這時 agent 才真正從「會講」變成「會做」。

Session 把一次工作變成可延續的上下文

這一輪的 prompt、最終回答、events、token 用量、tool invocation count 都會被寫回 session turn。下一次 resume 時，這些內容就能被濃縮成 continuation 摘要，重新回到 prompt 前面。

如果你要自己對另一個 AI 助手專案做 實況檢查

先找總入口

不管是 CLI、server handler 還是 app facade，先找最終會調到 agent 執行時 的那條主要流程。

再找 prompt 是在哪裡被組裝

如果整個專案都找不到明確 提示詞組裝，往往代表 repo instructions、commands、skills 之類能力還只是概念，而不是實際做出來層。

最後追工具操作影響與會話回寫

確認工具呼叫前有沒有 gate，工具結果有沒有寫回狀態，session 是否只是存檔，還是能回傳到下一輪 prompt。

這個 實況檢查 教我們什麼

1

Aixlarity 已經是能跑的 外殼，不只是 spec

CLI、provider adapter、工具迴圈、工作記錄保存 都有實作與測試，足以拿來當教學樣本。

2

完成度高低不是平均分布

有些層像 提示詞組裝 和 session 很完整；有些層像 container、memory、MCP 則更像正在補線中的第二曲線。

3

文件必須把主要流程與基本架構分開

否則讀者會把「有模組」誤讀成「有完整產品能力」，這就是 AI coding agent 文件常見的不準確來源。
