前言

AI 助手的「外殼」是什麼？

你可能會想說：AI 不是已經會寫程式了嗎，為什麼還需要 IDE 和 Harness？這一章先不講大名詞，我們先看它到底幫你管了哪些事。

先講直覺

Harness 可以先想成「包在 AI 模型外面的工作台」。模型負責想辦法，harness 負責把正確的檔案、工具、權限、紀錄和模型設定準備好。模型像引擎，harness 則像方向盤、煞車、儀表板和行車紀錄器。

為什麼這份文件改用 IDE 當主角

Harness 原本很抽象：提示詞怎麼組、工具能不能跑、工作紀錄怎麼存、證據怎麼留下，通常都藏在 runtime 裡。Aixlarity IDE 把它們變成看得到的介面：批准卡片是權限控制，Mission Control 是任務狀態，Diff Review 是審核門檻，Terminal Replay 和 Browser Recording 是證據。用 IDE 讀 harness，比只看 CLI 輸出更容易建立直覺。

第一次看？先別急

不用一次把所有名詞都背下來。建議看完這一章後，先看 IDE Harness Lab，用一個小任務把提示詞、工具、工件、工作紀錄這幾個基本概念看懂就好。剩下的名詞，後面讀到時自然會接起來。

為什麼要先講這個？

市面上介紹 AI 助手時，通常會強調「模型多強」、「支援多少工具」、「哪家 API」。但這只回答了「引擎有多強」，沒有回答「引擎是怎麼被操控的」。

舉個例子：AI 模型很會寫程式，但它自己不會以下這些事——

  • 不會自己去讀你的專案檔案（需要 read_file 工具）
  • 不會知道你的團隊有哪些規範（需要 AGENTS.md 設定檔）
  • 不會知道哪些操作需要先問你（需要權限控制）
  • 不會記住上次做了什麼（需要工作紀錄系統）

這些事情全部都是 harness 在做的。所以，一個 AI 助手好不好用，很大程度取決於它外面那層系統有沒有做扎實。

一個簡單的對比

同樣都是「幫我審查這個檔案」，有沒有 harness，結果完全不同：

情境 只有 AI 模型（像 ChatGPT 網頁版） 有 Harness 的 AI 助手
你說：「幫我看 src/app.rs」 模型只能根據你貼的片段猜，給你一般性的建議。 助手會自己去讀那個檔案，甚至搜尋相關檔案，再給出具體建議。
專案有規定要先跑測試 除非你每次都提醒，否則模型不知道。 這個規定寫在 AGENTS.md 裡，會自動加入提示詞。
模型想跑 cargo test 純聊天模型沒辦法真的執行指令，只能口頭建議你。 Harness 會把它變成工具呼叫，經過安全檢查後真的執行。
你想接著昨天的工作繼續 對話一斷，前面做過什麼就忘了。 工作紀錄會保存完整上下文，讓你可以接續、回放、甚至分叉。

AI 模型和 Harness 各負責什麼？

這件事 誰負責？ 為什麼
看不完整的資訊也能提出解法 AI 模型 這是 AI 模型最擅長的能力。
決定哪些專案設定可以載入 Harness 這涉及安全信任，不能讓 AI 自己決定。
控制哪些檔案可以改、哪些指令可以跑 Harness 這是安全管理，必須由系統限制。
切換不同 AI 模型（OpenAI、Gemini、Claude 等） Harness 不同模型的 API 格式不同，需要轉換。
把工具執行結果餵回 AI 做下一步決策 兩者合作 AI 負責思考要做什麼；Harness 負責真的去做、然後把結果送回來。
記錄和回放工作過程 Harness 這是除錯和學習的基礎。沒有記錄就沒辦法追查問題。

從最簡單到最完整：AI 助手的五個層次

一個 AI 助手不是一步到位的，而是一層一層往上加的：

第 1 層

單純問答

你問一個問題，AI 回答。就像用 ChatGPT 聊天。這還不算 Agent。

第 2 層

加上工具

AI 開始能「做事」了：讀檔案、跑指令、修改程式碼。系統會把工具的執行結果再餵回 AI，讓它決定下一步。這就是所謂的 agent loop。

第 3 層

加上安全控制

AI 能「做事」之後，就需要限制它「能做哪些事」。比如：讀檔可以自動做、但刪檔要先問你、跑測試可以但不能裝軟體。這一層包含 Trust（信任）、Sandbox（沙盒）、Permission（權限）。

第 4 層

加上專案知識

把你的專案規範（AGENTS.md）、任務模板（commands）、方法論（skills）、附件、歷史記錄，全部組合成一份完整的提示詞送給 AI。這就是 提示詞組裝。

第 5 層

可長期使用

工作紀錄可以回放和續接、可以切換不同 AI 模型、可以追蹤費用、可以檢查每次工具呼叫的結果。到了這一層，AI 助手才真正能融入日常開發流程。

簡單範例：同一個任務在不同層次的差異 
【第 1 層】單純問答
你：幫我審查 src/app.rs
AI：一般來說審查要注意錯誤處理、命名規範⋯⋯（一般性的建議）

【第 2 層】有工具
你：幫我審查 src/app.rs
AI → 呼叫 read_file("src/app.rs")
工具 → 回傳檔案內容
AI：第 42 行有個錯誤處理漏洞，建議改成⋯⋯（具體建議）

【第 3 層以上】有安全控制 + 專案知識
你：幫我審查 src/app.rs
系統：讀取 AGENTS.md（專案規範）
系統：設定 sandbox=只能改工作區, permission=需確認
AI → 呼叫 shell("cargo test")
系統：這個操作需要你確認，允許嗎？
AI：根據測試結果和程式碼內容，有兩個具體風險⋯⋯

沒有 Harness 會怎樣？

A

AI 看不到你的專案規範

沒有提示詞組裝，AI 不知道你的團隊怎麼工作、coding style 是什麼、哪些測試要先跑。

B

惡意專案可能注入指令

沒有信任機制，不明來源的專案可能在設定檔裡偷塞指令，被 AI 當成高優先規則執行。

C

換模型就要重寫一堆程式

沒有模型供應商轉接層，從 OpenAI 換成 Gemini 就要改一堆 API 呼叫格式、串流處理、錯誤處理。

D

出問題找不到原因

沒有工作紀錄，每次失敗都無法回放。你不知道問題出在提示詞、工具、模型，還是權限設定。

Aixlarity 為什麼適合用來學這些？

Aixlarity 不是世界上最完美的 AI 助手，但它有一條非常清楚的 Rust 程式碼路徑：從 CLI 入口 → 工作區偵測 → 信任檢查 → 提示詞組裝 → AI 對話迴圈 → 工作紀錄保存，每一步都看得到程式碼。

更重要的是，它同時展示了兩種學習價值：

  • 已完成的功能：CLI、提示詞組裝、工具系統、工作紀錄、模型供應商轉接⋯ 每個都有完整的程式碼和自動化測試。
  • 還在發展的功能：Container 沙盒、MCP 外部工具、Token 快取⋯ 可以讓你練習分辨「已完成的功能」和「規劃中的方向」。
建議的閱讀順序

如果你第一次看，接下來最順的順序是：IDE Harness Lab → 入門範例 → 名詞速查 → 架構全貌。先用 IDE 看見權限、工件、終端機證據，再回頭讀 Rust runtime，整個系統會清楚很多。
