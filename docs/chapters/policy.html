Hooks 和 Sub-Agent

Hooks、權限確認、Sub-agent——這三個機制讓你可以在 AI 執行操作之前介入，決定要不要放行。

先講結論

GCD 已經不只是「模型 + 工具」的直線系統。它另外有三種會改變執行路徑的 執行過程中的管理機制：permission prompt 決定要不要先問人、hooks 允許本地政策在工具前後插入、spawn_agent 則讓一段任務能被委派給有上限的 sub-agent。這三者加總起來，才構成比較接近產品等級的系統 的管理層。

先把這一章想成「誰可以插手 代理迴圈」

如果前面工具章主要在回答「模型能用哪些能力」，這一章就在回答另一個更接近 production 的問題：當模型已經決定要做某件事時，還有哪些層可以在最後一刻改變執行結果。這些層不是可有可無的，而是實際的 執行時 管理層面。

P1

Permission prompt

把高風險工具交還給人類拍板，處理的是「誰同意」。

P2

Hooks

在工具執行前後插入本地政策，處理的是「團隊想額外加什麼規則」。

P3

ToolContext

把 workspace、sandbox、hooks、coordinator provider / key / permission 帶進工具執行當下。

P4

spawn_agent

在主 代理迴圈 內開出一條有上限的委派路徑，處理的是「哪些工作值得切成子任務」。

主要流程上，這些政策插點插在哪裡

把 App::exec 與 run_agent() 串起來看，會發現 執行時 政策不是散落在各處，而是有清楚的插入時機。App::exec 先做 workspace / trust / providers / commands / skills / instructions / plugins 的前置載入；真正進到 run_agent() 後，才再把 MCP tools 與 hooks 載進來，並建立一個附帶 coordinator 資訊的 ToolContext。

教學性示意：Runtime 政策插點的進場順序 
App::exec
1. discover workspace / trust
2. load providers / commands / skills / instructions
3. load plugin catalog
4. assemble prompt
5. build AgentRunOptions
6. run_agent(agent_options, plugins.into_tools())

run_agent
7. load_mcp_tools(workspace_root)
8. HooksConfig::load(...)
9. build ToolContext { sandbox, hooks, coordinator_provider, ... }
10. provider response -> permission check -> pre-hook -> tool execute -> post-hook
11. if tool == spawn_agent -> run_agent(sub_options, Vec::new())

重點：
- plugin tools 在 App::exec 前段進來
- MCP 與 hooks 在 執行時 內部才進來
- sub-agent 不是另一個 App::exec，而是直接重走 run_agent()
這裡最值得注意的差別

spawn_agent 走的是 run_agent(sub_options, Vec::new())，不是回到 App::exec。因此它不會重新跑 commands / skills / instructions / plugin catalog 那段前置載入，但會重新載入 MCP tools 與 hooks。這個差別會直接影響 sub-agent 手上到底有哪些能力。

Hooks：真正的定位是本地政策注入，不只是記錄

很多文件提到 hooks 時，會把它寫成「可接日誌」或「可做整合」。這樣不算錯，但太輕了。從實作來看，GCD 的 hooks 真正價值在於：它允許團隊在不改每個工具實作的情況下，對工具的執行前後加上自己的政策。例如阻擋危險 shell pattern、在 patch 後觸發額外檢查、或把工具結果送到外部稽核流程。

Hook 點位 何時觸發 能不能阻擋 它適合做什麼
PreToolUse 工具真正執行之前 可以，exit 2 即 deny 政策檢查、風險攔截、命令審核、額外批准。
PostToolUse 工具執行完成之後 不可以 通知、稽核、同步外部系統、附加檢查。

Hook 載入路徑，有一個非常值得誠實寫出的 實況檢查

hooks.rs 裡的註解說的是「從 global config dir 與 workspace .gcd/hooks.json 載入」，檔頭註解甚至還提到舊的 .toml 路徑；但真正的 call site 在 agent.rs，傳進去的「global」路徑其實是 workspace_root/.config/gcd。換句話說，目前 hooks 的實際載入含義，比較像工作區相對的「類全域 + 工作區本地」兩層合併，而不是使用者 home 目錄下那種真正的全域設定。

層次 實際路徑 程式行為 判讀
類全域 hooks workspace_root/.config/gcd/hooks.json run_agent() 會先讀這裡，再讀 workspace hooks。 的現況，但不是典型使用者全域目錄含義。
workspace hooks workspace_root/.gcd/hooks.json 如果檔案存在，會 append 在前者之後，兩層都跑。
真正使用者全域 hooks ~/.config/gcd/hooks.json 之類 從目前 call site 看，主要流程沒有明確走到這條路徑。 的命名 / 文件含義。
註解漂移本身就是文件應該教的事

這就是 外殼工程 文件最值得誠實的地方之一。對讀者來說，重要的不是抓出「誰寫錯」，而是學會辨識：程式註解、配置命名、實際 call site 有時不完全一致，真正該相信的是執行路徑。所以，hooks 的「全域設定」目前只能保守描述。

PreToolUse 的退出碼，決定它到底是 gate 還是 advisory

只知道「PreToolUse 可以 deny」還不夠。真正影響完整功能的，是各種退出結果怎麼被 執行時 解讀。GCD 目前的行為相當清楚，但也有一個容易被忽略的細節：只有 exit 2 會真的阻擋工具；其他非零或 timeout 都只會留下 warning，工具仍繼續走。

Hook 結果 Runtime 反應 工具會不會繼續執行 這代表什麼
exit 0 Allow 會 hook 把這次工具呼叫放行。
exit 2 Deny 不會 這是明確阻擋工具的唯一硬 gate。
其他非零退出碼 Error 會 hook 失敗只會留下警告，不會自動 fail-closed。
timeout / 執行錯誤 Error 會 目前設計偏向 fail-open，而不是 fail-closed。
範例：同一個 shell call，兩種 hook 寫法差很多 
例子 A：嚴格阻擋
{
  "pre_tool_use": [
    {
      "command": "echo \"$GCD_TOOL_ARGS\" | grep -q 'rm -rf' && exit 2 || exit 0",
      "tools": ["shell"]
    }
  ]
}
=> 匹配到時，shell 真的不會執行

例子 B：只留下警告
{
  "pre_tool_use": [
    {
      "command": "echo 'suspicious shell' >&2; exit 1",
      "tools": ["shell"]
    }
  ]
}
=> 執行時 會印 warning，但 shell 仍會繼續跑

這代表：
hook 寫法不只是語法問題，而是直接決定政策是 fail-open 還是 fail-closed

Hook 比 plugin 更接近「真 shell」

這裡有個很容易被忽略、但非常值得拿來教學的差異。plugin tools 執行時，會先做 tokenization，再以 Command::new(program).args(...) 的方式執行；hooks 則是直接走 sh -c。因此兩者雖然都叫做「可擴充」，風險程度卻不同。plugin 比較像受控的命令封裝；hook 則更像把一段 shell 政策插進 runtime。

面向 Plugin tool Hook command
執行方式 Command::new(program).args(...) sh -c
用途 把一個本地能力包成新工具 在既有工具前後插入政策
資料來源 schema + command template GCD_TOOL_NAME、GCD_TOOL_ARGS、GCD_TOOL_RESULT
風險程度 偏向受控命令封裝 更接近直接 shell 政策

事件層的可以觀察性，目前還有一個盲點

從 agent events 看，無論是使用者在 permission prompt 按下拒絕，還是 hook 用 exit 2 擋下來，runtime 都只會送出同一種 ToolCallDenied 事件。換句話說，對 replay / JSONL 讀者而言，「被誰擋下來」這個拒絕來源目前不會進事件 schema。主程式會把 hook 拒絕寫成工具訊息內容，但如果你只看 events，仍會丟失原因。

教學性示意：同一個 denied event，可能來自不同來源 
情境 A：使用者拒絕 shell
AgentEvent::ToolCallDenied { tool_name: "shell", ... }

情境 B：PreToolUse hook exit 2
AgentEvent::ToolCallDenied { tool_name: "shell", ... }

差別只存在於：
- stderr 上的提示
- 給模型看的 tool error message

如果只看事件流：
你無法直接分辨 deny 是來自人工批准還是本地政策

這個盲點不代表 hooks 不可用，而是提醒我們：當一個 agent 開始引入更多 執行時 政策時，事件 schema 也要跟著成長。否則 debugging 會愈來愈倚賴 stderr 或人工推理，而不是結構化證據。

spawn_agent：它是 有限制的委派，不是完整多代理平台

GCD 的 spawn_agent 已經能把子任務交出去，但最誠實的理解方式不是「已經有完整多代理系統」，而是「已有一個帶上限的委派工具」。這個差異很重要，因為它決定你怎麼設計例子、怎麼讀 執行時 evidence，也決定文件能不能保持可信。

面向 目前行為 判讀
工具名稱 spawn_agent 已經是 built-in tool，可直加入主要流程。
預設回合數 5 代表它被設計成短任務委派，不是長程常駐工作代理。
最大回合數 10 有明確上限，避免委派無限膨脹。
最大深度 2 再下一層就會被硬擋，屬於 有限制的委派。
輸出模式 不串流 較像背景工作摘要，而不是完整互動工作台。
auto-git 強制關閉 避免子任務直接替主要流程做提交。
回傳形式 摘要 JSON 會回傳 status、turns_used、response、前 10 筆工具摘要與 token 用量。

Sub-agent 到底繼承了什麼，又失去了什麼

從 ToolContext 與 CoordinatorTool 的實作來看，sub-agent 會繼承不少父層執行條件，但也明確失去一些能力。這就是 教學範例 很值得拿來教的地方：委派不是「再開一個一模一樣的 agent」，而是「在有限條件下開一個新的 runtime」。

項目 sub-agent 是否繼承 說明
workspace root 會 仍在同一個工作區裡工作。
primary provider / API key 會 來自 ToolContext.coordinator_provider 與 coordinator_api_key。
permission level 通常會 主要流程會把父層 permission 放進 context；如果 context 沒填，才回退到 auto-edit。
sandbox 會 子代理仍跑在同一套 sandbox policy 下。
hooks 會重新載入 因為 sub-agent 直接再跑一次 run_agent()。
MCP tools 會重新載入 run_agent() 會再讀一次 workspace .gcd/mcp.json。
plugin tools 不會 spawn_agent 呼叫的是 run_agent(sub_options, Vec::new())，沒有把 plugin catalog 傳下去。
fallback providers 不會 CoordinatorTool 只用 with_defaults() 建 options，沒有補回父層 fallback list。
session / checkpoint persistence 不會是第一級能力 sub-agent 不經過 App::exec，因此不會自動走 checkpoint / session 存檔流程。
範例：父代理把 repo 掃描委派給 sub-agent 
父代理工具呼叫：
spawn_agent({
  "task": "掃描 src/ 目錄，列出最值得先 review 的三個檔案與原因",
  "max_turns": 4
})

sub-agent 真的會得到：
- 同一個 workspace
- 同一個 primary provider
- 同一套 sandbox
- 重新載入的 hooks
- 重新載入的 MCP tools

sub-agent 不會自動得到：
- 父層 plugin tools
- 父層 fallback provider list
- 獨立 checkpoint / session 記錄
- 串流到主終端的完整逐步輸出

所以它比較像：
一個受限的委派工作代理
不是完整平行工作平台

Hooks 與 sub-agent 放在一起看，會更像真正的管理層

把兩者分開看，你會覺得一個是 shell hook、一個是協調工具；放在一起看，才會發現它們其實都在回答同一件事：runtime 裡還有誰可以在模型之外重新塑形這條執行路徑。hook 影響的是工具前後的政策；sub-agent 影響的是任務分工的形狀。兩者都不是 prompt 文案本身，但都會改變最後發生的事。

G1

Hook 是局部政策

它在單一工具呼叫前後介入，讓團隊把管理規則插進 runtime。

G2

Sub-agent 是任務切分政策

它不是直接改單一工具，而是決定哪些工作值得交出去做。

G3

兩者都受限於可以觀察性

如果事件 schema 不記拒絕來源，或 sub-agent 沒有第一級 session 證據，debugging 仍會有斷點。

G4

兩者都和 trust 有邊界張力

workspace hooks 與 workspace MCP config 目前沒有和 commands / skills / providers 完全共用同一套 信任檢查。

實況檢查：這一層哪些能力成熟，哪些還需要保守描述

能力 判讀 說明
PreToolUse / PostToolUse lifecycle 兩種 hook 都已在主 執行時 生效，且帶有 timeout 與 tool-specific matching。
Hook deny 作為硬 gate exit 2 會真的擋下工具執行。
Hook trust / global path 含義 call site、註解、命名之間仍有落差，workspace-owned hooks 也未與其他 config 完全同 gate。
Bounded sub-agent delegation 有 max turns、max depth、provider / sandbox / permission 繼承與摘要回傳。
Sub-agent 作為第一級工作代理 已有事件與工具摘要，但沒有自動 checkpoint / 工作記錄保存，也不繼承 plugin / fallback 全貌。
結構化事件中的拒絕來源 ToolCallDenied 目前不區分人工拒絕與 hook 拒絕。

把這一章濃縮成一句話

真正的 外殼，不只會做事，也要會管理做事的方式

Hooks、permission prompt、sub-agent 這些層的存在，說明 GCD 已經開始從「能呼叫工具的 agent」走向「能管理安全、任務分配和本地規則的完整系統」。但這個部分也還沒走完，尤其是 trust 對齊、拒絕來源與 sub-agent 證據模型，都是下一步最值得持續補強的地方。
