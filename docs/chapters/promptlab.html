Prompt 解剖室：最終 Prompt 如何一層一層長出來

這一章不只講 prompt 來源，還要把真正的組裝順序、插入時機、限制條件與常見誤判分開來看。讀完後，你應該能自己重建 GCD 的 最終提示詞 形狀。

這一章在補什麼

前面的 Pipeline 與 提示詞來源 章，已經把來源和概念講清楚了；但如果你真的要回到原始碼，還是會卡在更具體的問題上，例如：commands 是先展開還是先注入檔案？!{...} 在哪一步被改寫？Session Continuation 是不是由 assemble_prompt() 直接產生？這一章的目的，就是把這些實作細節拼成一張完整地圖。

先講最重要的現況判讀

層次 目前在哪裡完成
commands / skills / instructions / file injection / shell block 組裝 prompt.rs 與 prompt/ 子模組
工作接續 回插到 最終提示詞 app.rs 透過 inject_section_before_task()
binary attachment 真正落入各 provider request attachments metadata 已存在，但 adapter 實際做出來程度不一致
自動 skill 選擇 / 自動 prompt routing 目前沒有主要流程 / 未實作

真正的組裝流程，其實分成兩段

很多文件會把 提示詞組裝 寫成一個單一步驟，但在 GCD 裡，這條線其實分成兩段。第一段是 assemble_prompt() 把 repo instructions、command、skill、attachments、shell approval 與 執行時 metadata 組成一份「基礎 最終提示詞」。第二段則是在 App::exec 裡，如果有 resume 或 fork，再把 # Session Continuation 插進 # Task 之前。

01

先解析使用者輸入是不是 custom command

parse_command_invocation() 只要看到輸入以 / 開頭，就會先把它當成 command invocation，拆出名稱、args 與完整原始命令。

02

把 command 模板先渲染成文字

如果符合 command，render_command_prompt() 會先加上 # Custom Command header，再做 {{args}} 置換或補上 Raw command。到這一步為止，還沒有做檔案注入與 shell block 改寫。

03

再展開 @{...} 附件注入

expand_file_injections() 會把文字檔、目錄、二進位附件轉成 prompt segment 與 attachment metadata。這一步之後，prompt 內部才真的出現 <injected-file> 或 <attached-file>。

04

再把 !{...} 改成 shell approval block

sanitize_shell_blocks() 只會作用在非 injected segment，避免把注入檔案裡原本存在的字串誤當成命令模板。這一步也會產生 pending_shell_commands。

05

最後才把各段 section 串成基礎 最終提示詞

順序固定為 # Runtime、# Repository Instructions、# Loaded Skill、# Task、# 待審核指令。

06

如果有 session reuse，再由 app.rs 插入 continuation

inject_section_before_task() 會把 # Session Continuation 插到 # Task 前面。這表示 session 不是在 prompt 子模組內生成，而是 app orchestration 最後補上的 執行時的上下文。

一個完整例子：從使用者輸入到 最終提示詞

假設使用者輸入的是下面這行，而且工作區已 trusted、選了 code-review skill、同時從舊 session resume：

教學性輸入例子 
/review src/app.rs

而對應 command 模板長這樣：

教學性 command 模板 
description = "Review a file with repo rules"
prompt = """
Use @{README.md} as project context.
Review target: {{args}}
If needed, run !{cargo test -p gcd-core}
"""

那麼這條線大致會經過下面幾個中間狀態：

步驟 中間結果長什麼樣子 這一步在解什麼問題
command render 後 會先出現 # Custom Command header 與 Review target: src/app.rs 把 workflow 模板檔案化，而不是直接把 user input 塞進 prompt。
file injection 後 @{README.md} 變成帶行號的 <injected-file> 把 repo 證據變成可控 preview，而不是任意全文貼上。
shell sanitize 後 !{cargo test -p gcd-core} 變成 <shell-approval> 把命令意圖和命令執行分開。
section 組裝後 會有 # Runtime、# Repository Instructions、# Loaded Skill、# Task 把來源分層，而不是做成一大塊無法追蹤的文字。
app 插 continuation 後 # Session Continuation 被放進 # Task 前面 在不重灌整段舊歷史的前提下，補回最近工作摘要。

Commands 不是直接塞模板，而是先加一層可追蹤 header

這個細節很值得拆出來講，因為許多 agent 只會把 command 模板靜默展開，事後很難知道 prompt 中哪些文字其實是某個命令模板帶進來的。GCD 則會明確插入 # Custom Command、名稱、描述、來源路徑與工作區。這讓 prompt 事後可追蹤，也讓 session replay 更容易看懂。

P1

命令名可追蹤

你能知道這輪到底是 /review 還是 /review:security 在主導任務。

P2

來源路徑可追蹤

當模板行為怪異時，你可以直接回頭看那個 TOML 檔，不必猜是哪段 prompt 生出來的。

P3

{{args}} 與 Raw command 保住參數資訊

即使模板沒留 placeholder，使用者參數也不會無聲消失。

@{...} 先展開、!{...} 後改寫，順序不能搞反

GCD 先做 file injection，再做 shell block sanitize。這個順序不是偶然，而是為了避免把注入檔案裡原本存在的 !{...} 內容誤當成 command template 的待執行 shell。換句話說，只有原始 prompt 文字本身的 shell block 會被改寫，注入檔案內容則被當成證據保留。

情境 如果順序做錯會怎麼樣 GCD 目前怎麼避免
README 裡剛好寫了 !{rm -rf ...} 當示例 如果 sanitize 太早或太廣，可能把注入檔案內容也誤判成待批准命令。 只對非 injected segment 做 shell block 改寫。
prompt 模板本身寫了 !{cargo test} 如果不 sanitize，就會把命令意圖直接當普通文字，失去批准節奏。 轉成 <shell-approval> 並記入 pending list。
範例：注入檔案中的 !{...} 不會被當成待批准命令 
guide.txt 內容：
請不要真的執行 !{cargo clean}

prompt 模板：
Use @{guide.txt}
Then run !{cargo test -p gcd-core}

結果：
- guide.txt 中的字樣保留為注入證據
- 只有模板裡的 cargo test 變成 shell-approval block

File injection 的路徑解析其實也有規則

resolve_injected_path() 不是把任何路徑都照單全收。相對路徑會先嘗試從 workspace.current_dir 找，再退到 workspace.root；絕對路徑則必須落在工作區根或目前目錄底下，否則會被拒絕。這表示 injection 不只是便利語法，也是工作區邊界的一部分。

輸入 目前解析策略 對文件讀者的意義
@{src/app.rs} 先找 current_dir，再找 workspace root 相對路徑可以跟著你當前工作位置解析。
@{/absolute/path/in/workspace} 如果位於 workspace 內，可接受 允許顯式絕對路徑，但仍受工作區限制。
@{/etc/passwd} 拒絕 防止 prompt injection 直接把工作區外檔案拉進來。

大小限制本身就是 prompt 設計的一部分

GCD 對文字注入有 400 行與 32KB 的上限，目錄 listing 則上限 200 個檔案。這些數字看起來只是常數，但其實反映了一個重要判斷：提示詞組裝 不能只追求「把更多東西塞進去」，還要顧及 provider 成本、上下文膨脹與可讀性。這也是為什麼 prompt engineering 和 外殼 工程 不該分開看。

L1

400 行上限

避免注入單一大檔時把 prompt 直接撐爆。

L2

32KB 上限

即使每行都不長，也要從總 byte 數控制膨脹。

L3

200 個目錄項目上限

目錄注入重點在建立概覽，不是把整棵樹全部展平。

Skills 在這條線裡是顯式加載，不是自動召回

assemble_prompt() 只會在 selected_skill 明確存在時，才加進 # Loaded Skill section。這個設計很保守，但也很可教。因為讀者可以清楚知道：某輪 prompt 為什麼有 skill，是因為執行時明確選了它，而不是系統悄悄做了某種難以追溯的 prompt routing。

這是一個刻意的取捨

自動召回技能並不是不能做，但一旦做了，就必須額外處理可解釋性、衝突排序與除錯成本。GCD 目前選擇先把 skill 當成顯式輸入，這很符合它作為教學用的範例系統 樣本的定位。

Session Continuation 不在 assemble_prompt() 裡，這點很重要

這是很多讀者最容易忽略的地方。assemble_prompt() 組完後，App::exec 會根據 resume / fork 狀態，另外呼叫 render_session_context() 生成 continuation，再用 inject_section_before_task() 把它插到 # Task 前面。換句話說，session 不是 prompt module 的內部常駐成分，而是 app orchestration 依情境追加的 執行時的上下文。

如果你想問的是 答案 應該去哪裡看
基礎 最終提示詞 section 順序是什麼 看 prompt.rs assemble_prompt()
resume / fork 為什麼會多一段 session 摘要 因為 app.rs 在 prompt 組完後又插了一段 render_session_context() + inject_section_before_task()
為什麼有些 prompt 看起來比 pipeline 圖多一塊 通常就是因為你正在看的是帶 continuation 的最終結果 先對照 session reuse mode

最值得注意的幾個 實況檢查

目前只支援單一 selected skill

沒有多 skill 排序與合併主要流程，所以不要把它寫成已成熟的 skill orchestration。

binary attachment 會形成 metadata，但不等於各 provider 都完整吃得到

prompt 層已經保留了 BinaryReference，但從這裡到各 adapter 的實際做出來仍需保守標 Implemented。

工作接續 是 app 層追加，不是 prompt 模組原生 section

這個差異會影響你怎麼 debug prompt，也會影響你未來如果想抽象化 提示詞組裝器 的邊界。

shell pending list 是從模板文字抽出，不是模型自己提議的 shell

它代表 workflow 預先埋好的命令意圖，不該和 執行時 tool call 混成同一件事。

如果你要自己重建一輪 prompt，最推薦的順序

01

先看 user input 有沒有命中 command

這會決定你一開始是在處理 plain task，還是先有一層 # Custom Command 模板。

02

再看 trust 與 selected skill

這兩件事會決定 # Repository Instructions 與 # Loaded Skill 是否存在。

03

再看有沒有 @{...} 與 !{...}

這會直接影響 attachments 與 待審核指令s。

04

最後再確認有沒有 resume / fork

如果有，才會在最後多出一段 continuation；如果沒有，就不要把 session context 腦補進所有 prompt。

把這一章濃縮成一句話

Prompt 不是一個字串，而是一條有順序的編輯管線

GCD 的 最終提示詞 不是把資料隨便串起來，而是先經過 command render、file injection、shell sanitize、section 組裝，再由 app 層視情況插入 工作接續。當你能把這條順序重建出來，你就不只是「看過 prompt」，而是真的開始理解 外殼 如何把工作流寫進模型上下文。
