IDE

Aixlarity IDE：看得見過程的 AI 工作台

Aixlarity IDE 基於 VS Code fork，但目標不是複製一個聊天側邊欄，而是把 agent harness 的任務控制、證據、審核、瀏覽器、終端機、模型供應商和編輯器動作做成可操作、可教學的產品介面。

一句話定位

Aixlarity IDE 是 AI agent 的控制台：TypeScript 前端負責讓人看懂、批准、審核；Rust 背景服務負責 agent loop、模型供應商轉接、工具權限、工作紀錄與任務狀態。兩邊用 JSON-RPC over Electron IPC 連接。

為什麼 IDE 是教 Harness 最好的入口

CLI 很適合展示 agent runtime，但很多 harness 問題在 CLI 裡不容易看見：人類何時批准？AI 每輪改了什麼？瀏覽器到底看到了什麼？終端機輸出怎麼附回任務？多個長任務如何恢復？IDE 把這些問題變成具體畫面。

你想知道的事 IDE 讓你看哪裡 後端負責什麼
這個 AI 任務現在做到哪裡？ Mission Control 任務卡 mission_control/load / mission_control/save
AI 這輪到底改了什麼？ 工件檢視器 + Visual Diff Review artifacts/list / artifacts/review
它真的跑過測試或看過頁面嗎？ Terminal Replay + Browser Recording shell transcript、browser evidence、audit log
高風險操作要怎麼先問人？ 批准卡片 permission gate、approval request / response
模型要怎麼切才不亂？ Provider Control Center scoped provider registry、API key env、model validation
游標、選取文字、問題面板怎麼交給 agent？ hover / Problems / selection actions ide_context + agent prompt assembly

核心能力：先抓住五件事

P0

Mission Control

多工作區、多任務、多 agent 的控制台。支援暫停、繼續、取消、重試，並把任務時間線、批准、工件和審計合在一起。

P0

Artifact System

實作計畫、任務清單、diff、測試報告、截圖、瀏覽器錄製、終端機紀錄都是結構化工件，可留言、批准、退回、匯出證據。

P0

Integrated Browser Agent

瀏覽器證據不只截圖。它保留 DOM、console、network、截圖、影片與操作時間線，讓瀏覽器狀態能回到提示詞與審核流程。

P0

Terminal Manager / Replay

每個指令都有負責任務、cwd、env 摘要、stdout、stderr、exit code、耗時，可以附到工件；危險命令需要批准。

P0

Editor-native Agent Actions

Diagnostic hover 的 Explain and fix、Problems panel 的 Send all to Agent、選取文字與終端機輸出送給 agent，讓 agent 進入日常開發位置。

Mission Control：從聊天窗變成任務控制台

你可以先把 Mission Control 想成「AI 任務的看板」。任務不再只是對話紀錄，而是一個有狀態、有時間線、有工件、有批准佇列的工作單元。

區塊用途教學價值
Tasks 顯示 running、paused、waiting_review、failed、completed 任務。 學會把 agent work 建成狀態機，而不是一串聊天訊息。
Pending Approvals 集中處理 shell / write / patch 等高風險操作。 學會 human-in-the-loop 的產品位置。
Evidence 所有工件的審核佇列與檢視器。 學會把 AI 輸出變成可驗收工件。
Audit 記錄批准、審核、討論串、匯出等事件。 學會讓 agent 行為可追責。

Visual Diff Review：每輪 AI 修改都要能看懂

Aixlarity 參考 JuxtaCode / Meld 的思路，把 AI 每次修改做成細緻的視覺化 diff，而不是只顯示一段 patch。這是教學上很重要的一點：AI coding agent 的品質取決於「人能不能快速確認它改了什麼」。

Side-by-side / Unified 切換

同一個 diff 可以用兩種視角檢查，適合不同類型的 reviewer。

Compare Rounds

比較第 N 輪與第 N+1 輪 AI 修改，幫助使用者確認 agent 是否朝正確方向收斂。

逐段審核

每一段修改可以批准、留言、複製證據，讓審核不必只有整份批准或整份退回。

審核門檻

如果仍有未解決討論或未審核段落，工件不應該被草率標為 ready。

Browser Agent：把瀏覽器操作變成 evidence

「AI 開網頁看過了」不是可驗證證據。Aixlarity 的瀏覽器工件會把 DOM、console、network、截圖、影片、操作時間線整理成工件，讓 reviewer 能知道 agent 實際看到什麼、點了什麼、是否有前端錯誤。

Browser evidence 包含什麼 
browser_recording artifact
- action timeline
- current URL and title
- DOM snapshot / text summary
- console events
- network requests
- screenshot
- optional video
- policy: allowed domains / blocked domains / capture switches

Terminal Replay：測試不是口頭宣稱

開源 agent 尤其需要可追溯。Terminal Replay 會記錄命令的負責任務、cwd、env 摘要、stdout、stderr、exit code、耗時。當 agent 說「測試通過」，IDE 裡應該能打開對應紀錄。

欄位為什麼重要
command + cwd 同一個指令在不同路徑下結果可能完全不同。
stdout / stderr 模型摘要可能漏掉 warning 或 partial failure，原始輸出要保留。
exit_code 最基本的機械驗收，不能只看自然語言。
risk reasons 危險命令需要讓使用者知道為什麼要批准。

Provider Control Center：多供應商不是雜湊設定

Aixlarity 的 provider 管理吸收了 cc-switch 類工具的核心精神：切換要快，但設定要有秩序。IDE 支援工作區 / 個人 scope、presets、匯入匯出 bundle、model required、API key env 管理，並避免把 raw API key 匯入專案。

Scope

User / Workspace

個人 secret 留在 user scope，團隊可共享的 provider 設定形狀放 workspace scope。

Model

API provider 必填模型

避免「同 provider 不同模型」造成不可重現的 agent 行為。

Bundle

安全匯入匯出

分享 provider 設計，不分享 API key。這是開源版本特別重要的底線。

IDE Context：讓 agent 不靠使用者重述世界

IDE 會把目前檔案、游標、選取文字、diagnostics、Problems panel、終端機輸出等上下文送給背景服務。這讓 agent 可以處理「這個錯誤」、「這段選取」、「這個 terminal output」這類自然工作流程，而不是要求使用者手動貼一堆資訊。

Submission Readiness：文件承諾要有測試背書

這個 IDE 的產品品質不只靠手動檢查。專案包含多層 acceptance / readiness checks，確保功能不是文件上的願望。

驗收檢查內容
npm run test-aixlarity-quality CI-safe product gate：P1/P2、source readiness、provider readiness、docs 首頁品質。
npm run test-aixlarity-p1 P1 control-plane、provider、artifact、browser/terminal policy、studio 能力的靜態能力檢查。
npm run test-aixlarity-p2 後續 UX / product-quality capability groups。
npm run test-aixlarity-ui 實際啟動 Electron，操作批准卡片、Fleet、工件檢視器、diff review、逐段批准。
npm run test-aixlarity-submission Release gate：先跑 product quality，再檢查 Electron artifact、Info.plist、URL protocol、release agent binary。

把 IDE 當教材的講法

最好的 demo 順序是：送出一個小任務 → 批准卡片 → 終端機紀錄 → code diff 工件 → 逐段審核 → 批准工件 → 匯出證據。這一輪能一次教完權限、工具執行、證據、工件審核、人類批准與可持久化狀態。
