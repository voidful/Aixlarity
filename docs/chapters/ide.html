IDE

Aixlarity IDE：開源版 Antigravity 的 Harness 工作台

Aixlarity IDE 基於 VS Code fork，但目標不是複製一個聊天側欄，而是把 agent harness 的任務控制、證據、審核、瀏覽器、終端機、provider、editor actions 全部做成可教學的產品表面。

一句話定位

Aixlarity IDE 是 Harness Control Surface：TypeScript 前端負責 human-in-the-loop 操作與視覺化證據，Rust daemon 負責 agent loop、provider adapter、tool permission、session / mission persistence。兩邊用 JSON-RPC over Electron IPC 連接。

為什麼 IDE 是教 Harness 最好的入口

CLI 很適合展示 agent runtime，但很多 harness 問題在 CLI 裡不容易看見：人類何時批准？AI 每輪改了什麼？瀏覽器到底看到了什麼？終端機輸出怎麼附回任務？多個長任務如何恢復？IDE 把這些問題變成具體 UI。

Harness 問題 IDE 表面 後端責任
任務如何被建模 Mission Control task cards mission_control/load / mission_control/save
AI 修改如何被審核 Artifact Inspector + Visual Diff Review artifacts/list / artifacts/review
工具執行如何留下證據 Terminal Replay + Browser Recording shell transcript、browser evidence、audit log
高風險操作如何批准 Approval Cards permission gate、approval request / response
多模型如何安全切換 Provider Control Center scoped provider registry、API key env、model validation
編輯器上下文如何送給 agent hover / Problems / selection actions ide_context + agent prompt assembly

P0：開源 Antigravity 必須有的五個核心面

P0

Mission Control

多 workspace、多 task、多 agent 的控制台。支援 pause / resume / cancel / retry，並把 task timeline、approval、artifact 和 audit 合在一起。

P0

Artifact System

Implementation Plan、Task List、Diff、Test Report、Screenshot、Browser Recording、Terminal Transcript 都是結構化 artifact，可留言、approve / reject、export evidence。

P0

Integrated Browser Agent

Browser evidence 不只截圖。它保留 DOM、console、network、screenshot、video 與 action timeline，讓 browser state 能回到 prompt 和 review。

P0

Terminal Manager / Replay

每個 command 都有 ownership、cwd、env 摘要、stdout、stderr、exit code、duration，可以附到 artifact，危險命令需要 approval。

P0

Editor-native Agent Actions

Diagnostic hover 的 Explain and fix、Problems panel 的 Send all to Agent、selection / terminal output send to agent，讓 agent 進入日常開發位置。

Mission Control：從聊天窗變成任務控制台

Fleet 以前比較像 session browser；現在 Mission Control 的重點是任務編排。任務不再只是對話紀錄，而是一個有狀態、有 timeline、有 artifacts、有 approval queue 的工作單元。

區塊用途教學價值
Tasks 顯示 running、paused、waiting_review、failed、completed 任務。 學會把 agent work 建成狀態機，而不是一串聊天訊息。
Pending Approvals 集中處理 shell / write / patch 等高風險操作。 學會 human-in-the-loop 的產品位置。
Evidence 所有 artifact 的 review queue 與 inspector。 學會把 AI 輸出變成可驗收工件。
Audit 記錄 approval、review、thread、export 等事件。 學會讓 agent 行為可追責。

Visual Diff Review：每輪 AI 修改都要能看懂

Aixlarity 參考 JuxtaCode / Meld 的思路，把 AI 每次修改做成細緻的 visual diff review，而不是只顯示一段 patch。這是教學上很重要的一點：AI coding agent 的品質取決於「人能不能快速確認它改了什麼」。

Side-by-side / Unified 切換

同一個 diff 可以用兩種視角檢查，適合不同類型的 reviewer。

Compare Rounds

比較第 N 輪與第 N+1 輪 AI 修改，幫助使用者確認 agent 是否朝正確方向收斂。

Hunk Review

每個 hunk 可以 approve、comment、copy evidence，讓 review 不必只有整份 approve / reject。

Review Gate

如果仍有 open thread 或未審核 hunk，artifact 不應該被草率標為 ready。

Browser Agent：把瀏覽器操作變成 evidence

「AI 開網頁看過了」不是可驗證證據。Aixlarity 的 browser artifact 會把 DOM、console、network、screenshot、video、action timeline 整理成 artifact，讓 reviewer 能知道 agent 實際看到什麼、點了什麼、是否有前端錯誤。

Browser evidence 包含什麼 
browser_recording artifact
- action timeline
- current URL and title
- DOM snapshot / text summary
- console events
- network requests
- screenshot
- optional video
- policy: allowed domains / blocked domains / capture switches

Terminal Replay：測試不是口頭宣稱

開源 agent 尤其需要可追溯。Terminal Replay 會記錄命令的 ownership、cwd、env 摘要、stdout、stderr、exit code、duration。當 agent 說「測試通過」，IDE 裡應該能打開對應 transcript。

欄位為什麼重要
command + cwd 同一個指令在不同路徑下結果可能完全不同。
stdout / stderr 模型摘要可能漏掉 warning 或 partial failure，原始輸出要保留。
exit_code 最基本的機械驗收，不能只看自然語言。
risk reasons 危險命令需要讓使用者知道為什麼要批准。

Provider Control Center：多供應商不是雜湊設定

Aixlarity 的 provider 管理吸收了 cc-switch 類工具的核心精神：切換要快，但設定要有秩序。IDE 支援 workspace / user scope、presets、import/export bundle、model required、API key env 管理，並避免把 raw API key 匯入專案。

Scope

User / Workspace

個人 secret 留在 user scope，團隊可共享的 provider shape 放 workspace scope。

Model

API provider 必填模型

避免「同 provider 不同模型」造成不可重現的 agent 行為。

Bundle

安全匯入匯出

分享 provider 設計，不分享 API key。這是開源版本特別重要的底線。

IDE Context：讓 agent 不靠使用者重述世界

IDE 會把 active file、cursor、selection、diagnostics、Problems panel、terminal output 等上下文送給 daemon。這讓 agent 可以處理「這個錯誤」、「這段選取」、「這個 terminal output」這類自然工作流，而不是要求使用者手動貼一堆資訊。

Submission Readiness：文件承諾要有測試背書

這個 IDE 的產品品質不只靠手動檢查。專案包含多層 acceptance / readiness checks，確保功能不是文件上的願望。

驗收檢查內容
npm run test-aixlarity-quality CI-safe product gate：P1/P2、source readiness、provider readiness、docs 首頁品質。
npm run test-aixlarity-p1 P1 control-plane、provider、artifact、browser/terminal policy、studio 能力的靜態能力檢查。
npm run test-aixlarity-p2 後續 UX / product-quality capability groups。
npm run test-aixlarity-ui 實際啟動 Electron，操作 approval、Fleet、artifact inspector、diff review、hunk approval。
npm run test-aixlarity-submission Release gate：先跑 product quality，再檢查 Electron artifact、Info.plist、URL protocol、release agent binary。

把 IDE 當教材的講法

最好的 demo 順序是：送出一個小任務 → approval card → terminal transcript → code diff artifact → hunk review → approve artifact → export evidence。這一輪能一次教完 permission、tool execution、evidence、artifact review、human approval 與 durable state。
