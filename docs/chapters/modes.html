Modes

Plan、Live、REPL、JSON 與 JSONL：的不同使用方式

Aixlarity 有好幾種使用方式：Plan 模式、Live 模式、REPL、JSON 輸出等。每種方式看到的東西不一樣，適合不同的使用情境。

先講最重要的結論

Aixlarity 不是為每一種使用方式各寫一套系統，而是盡量讓同一份 AppOutput、同一條 App::exec 主要流程，對應到文字、JSON、JSONL、REPL、plan、live 這幾種讀法。因此，很多看似「只是介面」的選項，其實都在表達你要從 外殼 看見什麼。

為什麼模式層值得單獨成章

如果你把 AI 助手 只當成「打一段任務，等模型回來」的工具，那 CLI 旗標、REPL、JSONL 輸出看起來像附加功能；但如果你把它當成可教、可以觀察、可自動化的 外殼，這些就會變成核心設計。因為工程團隊通常同時需要四種視角：

M1

人類閱讀

看 prompt preview、assistant response、runtime summary，快速掌握一輪執行大意。

M2

腳本整合

讓 CI、外部工具、其他腳本能讀出結構化資料，而不是只能 parse 純文字。

M3

事件追蹤

把一次執行分成事件序列，讓 replay、除錯與教學可以對齊到同一套 執行時 語言。

M4

互動工作流

REPL、plan mode、resume / fork 讓 agent 不再只是一次性請求，而是持續工作系統。

先把模式分成四條軸看

Aixlarity 的「模式」不是只有 plan 與 live。比較精準的看法，是把它分成四條互相交錯的軸：

軸 選項 它在回答什麼
互動介面 CLI 子命令 / REPL 你是一次執行，還是要在終端裡持續工作？
執行模式 plan / live / replay 你要看規劃快照、真的跑 agent，還是回頭讀既有事實？
輸出序列化 text / JSON / JSONL 你是要給人看、給機器吃，還是要看事件流？
回應呈現 stream / no-stream 模型回覆要不要在終端逐步吐出，還是等整段完成再顯示？

把這四條軸分開之後，你會比較容易理解：--plan、--jsonl、--no-stream 並不是在切換三個互斥產品，而是在改變同一條 外殼 的觀察方式。

沒有 subcommand 時，CLI 其實會直加入 REPL

aixlarity 如果沒有帶任何 subcommand，不是報錯，而是直加入互動模式。這個細節很重要，因為它說明 Aixlarity 的預設基本概念更接近「終端裡的工作環境」，而不只是一次性命令。

範例：REPL 會怎麼啟動 
$ aixlarity

╔══════════════════════════════════════════════╗
║       Aixlarity Interactive Mode      ║
║    Type your task, or /help for commands       ║
╚══════════════════════════════════════════════╝

[先顯示 workspace overview]
aixlarity>

REPL 內建幾個 slash command，像 /overview、/providers、/trust、/sessions；其他任何輸入都會被直接當成一個 exec task。這種設計很 terminal-first，也和 Gemini CLI / Codex 類型產品的互動節奏比較接近。

對腳本作者的一個提醒

--json 或 --jsonl 並不會改變「沒有 subcommand 就進 REPL」這件事。換句話說，如果你要做真正穩定的 headless / machine-readable 呼叫，最好明確使用 aixlarity overview、aixlarity exec、aixlarity sessions replay 這類 subcommand，而不是單獨執行 aixlarity --json。

REPL 不是完整 CLI 的另一個外皮，而是帶預設值的互動工作流

如果你往下看 run_repl()，會發現 REPL 對一般任務輸入其實套了很明確的預設：persist_session = true、print_prompt = true、permission = auto-edit、stream = true、plan_only = false。這代表 REPL 比較像「為互動工作優化過的一組執行選項」。

在 REPL 直接輸入任務時 預設值 對使用體驗的意思
工作記錄保存 開啟 互動工作預設可延續、可 replay。
prompt preview 開啟 每輪都偏向可教、可檢查。
permission auto-edit 互動使用偏向務實，不是最保守也不是全自動。
streaming 開啟 回答會逐步出現在終端，比較像對話。
checkpoint / plan 關閉 REPL 預設偏向直接工作，而不是先做規劃快照。
這代表什麼

如果你需要更精細地控制 --plan、--checkpoint、--permission、--no-stream、--no-session，最清楚的方式仍然是使用 aixlarity exec。REPL 比較像一條意見明確、方便持續工作的預設路徑。

--plan 不是另一個 agent，而是同一條 App::exec 的分支

這是整章最值得講清楚的 實況檢查 之一。很多文件一看到 規劃模式，就會直覺把它寫成「另一套推理 agent」；但 Aixlarity 目前的 plan_only 不是這樣。它會走同樣的 workspace / trust / provider / catalog / prompt 組裝前半段，只是在真正建立 AgentRunOptions 前就分支返回，不會進 run_agent()。

01

plan 與 live 共享前半段

兩者都會先做 workspace discovery、trust evaluation、provider / commands / skills / instructions / plugins 載入，然後組出同一份 PromptAssembly。

02

plan 可以一樣保存 checkpoint 與 session

如果有開 --checkpoint 或 工作記錄保存，plan mode 也會把資料寫下來，只是 turn 的 mode 會標成 plan。

03

plan 不會呼叫 provider，也不會跑 工具迴圈

因為根本沒有進 run_agent()，所以沒有 provider call、沒有 assistant response、沒有 tool invocation，也沒有 live token 用量。

04

因此它更像規劃輸出模式，而不是獨立 runtime

這個判讀很重要，因為它會直接影響你怎麼寫文件、怎麼設計教學、怎麼預期 plan mode 的事件含義。

範例：同一個任務在 plan 與 live 下會差多少 
# Plan
aixlarity exec --plan --checkpoint "review src/app.rs 並列出修補方向"

你會得到：
- Execution plan
- Prompt preview
- Checkpoint saved
- 可能的 plan session turn
- events 只到 ExecutionPrepared / CheckpointSaved / SessionPersisted

# Live
aixlarity exec "review src/app.rs 並列出修補方向"

你會得到：
- Execution result
- Provider called
- AssistantMessage
- ToolCallRequested / Completed
- RunCompleted
- Runtime summary

文字、JSON、JSONL 不是同一份東西換格式而已

Aixlarity 的輸出層雖然都包在同一個 AppOutput 裡，但三種 render 其實在回答不同問題。

輸出型態 你會拿到什麼 最適合的用途
text 給人類閱讀的摘要、prompt preview、assistant response、runtime summary。 終端互動、教學、快速檢查。
JSON 完整的結構化物件，例如 exec 模式下的 ExecJson。 腳本整合、一次拿完整摘要。
JSONL 逐行事件或紀錄，用來做 event stream 與後處理。 批次處理、事件管線、後續 observability。

其中最容易被誤會的是 JSONL。在 exec 路徑下，render_exec_output() 目前輸出的 JSONL 其實是 execution events 的逐行序列，不是完整的 exec summary 被分成多行。這代表如果你要拿到完整 prompt、attachments、result.final_response 等完整摘要，--json 反而更直接。

情境 比較適合 原因
我想一次拿到完整 exec 摘要 --json 會直接得到完整 ExecJson 結構。
我想追事件流，或把事件逐行送去別的系統 --jsonl 每行就是一個 event payload。
我想手動看 prompt 與回答 預設 text 人類閱讀成本最低。

一個很值得文件化的細節：--no-prompt 只影響文字輸出

在 exec 路徑裡，print_prompt 只控制人類可讀的 text render 是否顯示 Prompt preview；但在 JSON 結構裡，prompt 欄位仍會被放進 ExecJson。這對 automation 很有價值，也正是文件需要特別點出來的地方。

範例：--no-prompt 的真正影響範圍 
aixlarity exec --no-prompt --json "review src/app.rs"

文字輸出：
- 不顯示 Prompt preview

JSON 輸出：
{
  "mode": "live",
  "prompt": "...完整 最終提示詞 仍在這裡...",
  ...
}

所以如果你的需求是「終端畫面不要太長，但機器還是要拿到完整 prompt」，Aixlarity 目前其實已經可以做到。

--jsonl 在 exec 與 replay 下的含義也不完全相同

這是另一個很適合教學的細節。exec 的 JSONL 比較像單輪執行事件流；sessions replay --jsonl 則會把每個 event 包成帶有 session_id、turn_index、turn_mode、input 的 wrapper 物件。換句話說，兩者都叫 JSONL，但服務的分析任務不同。

命令 JSONL 每行大致長什麼樣子 適合做什麼
aixlarity exec --jsonl ... 單個 AgentEvent 即時或近即時地消費執行事件。
aixlarity sessions replay --jsonl ... 帶 session_id / turn_index / payload 的事件包裝 離線分析既有 session、做 replay pipeline。

Streaming 是呈現策略，不是結果格式

--no-stream 也很容易被誤解成「不要用 SSE」或「不要有 events」。更精準的說法是：它主要改變 provider adapter 與終端的回應呈現方式。streaming 關閉後，OpenAI / Gemini / Anthropic adapter 會走非 streaming 路徑，但整體 ExecJson、session turn、runtime summary 的含義並沒有變成另一套格式。

選項 主要改變什麼 不會改變什麼
streaming = true 回答在終端逐步吐出，adapter 走 SSE 或對應 streaming 協定。 提示詞組裝、permission、工作記錄保存 的主要流程不變。
streaming = false 等整個 response 完成後再輸出。 最終 result 的結構、events 類型與 session mode 不因此變成另一種產品。

如果同時開 --json 和 --jsonl，哪個會贏

在目前 CLI 實作裡，輸出順序是先檢查 jsonl，再檢查 json。換句話說，如果兩者同時為真，實際上會輸出 JSONL。這不是最核心的功能，但對腳本穩定性來說，是值得文件化的小細節。

範例：輸出選項的優先順序 
if cli.jsonl {
    println!("{}", output.render_jsonl());
} else if cli.json {
    println!("{}", output.render_json());
} else {
    println!("{}", output.render());
}

用三個實務情境理解這一章

S1

我要先做教學式規劃，再人工確認

用 aixlarity exec --plan --checkpoint --json。這樣可以同時保存 prompt、checkpoint 與完整結構化規劃輸出。

S2

我要把事件送去別的分析系統

用 aixlarity exec --jsonl 或 aixlarity sessions replay --jsonl。前者看 live event，後者看既有 session 事件。

S3

我要在終端裡持續工作，不想每次都重打一堆旗標

直加入 REPL，接受它偏向互動工作的預設值；如果要更精準控制，再切回完整 CLI。

實況檢查：最值得記住的五件事

--plan 不是另一個 agent runtime

它是 App::exec 的前半段分支，會組 prompt、可存 session / checkpoint，但不進 run_agent()。

REPL 是有意見的預設工作流

它預設會保存 session、顯示 prompt、使用 auto-edit、開啟 streaming。

--json 比 --jsonl 更適合拿完整摘要

exec 的 JSONL 目前主要是事件流，不是完整 result object 的逐行拆分。

--no-prompt 只隱藏文字版 prompt

JSON 結構裡的 prompt 欄位仍然存在。

Streaming 改的是呈現節奏，不是 外殼 結構

關閉 streaming 後，主要流程仍是同一份 prompt、同一套 events、同一份 session 模型。

把這一章濃縮成一句話

模式層的價值，在於讓同一條 外殼 同時服務人與機器

Plan / live 決定你是在看規劃還是在看執行，REPL / CLI 決定你是持續工作還是一次觸發，text / JSON / JSONL 決定你是給人看還是給系統吃，stream / no-stream 則決定終端裡的節奏。把這些層講清楚，你就會知道 Aixlarity 不只是「能跑的 agent」，而是一個開始具備操作面設計的 外殼。
