Provider Lab

同一個任務，四種 Provider 協定如何變形

同一個任務，用 Gemini、OpenAI、Anthropic 分別跑，API 請求長什麼樣？這一章讓你直接看到差異。

先講這一章要解的困惑

很多人知道 Aixlarity 支援 Gemini、OpenAI、Anthropic，也知道有 Responses 與 Chat 兩種 OpenAI 路徑，但真正接到原始碼時，還是不知道這些差異到底落在哪裡。這一章的目標，就是把 provider layer 從「名詞列表」變成「可以看到的 請求格式調整 實例」。

先建立一個很重要的觀念

對 外殼 工程來說，新增 provider 並不只有一種工作量。只是新增一個 profile，和真正新增一條新協定，是完全不同層級的事情。前者比較像設定擴充；後者則會牽涉 request body、tool declaration、tool result 回送、streaming parser、usage 欄位與 fallback 含義。

如果你要新增的是 主要工作 工程難度
既有協定上的新 profile 補 profile、key env、base URL 與模型名稱。 低到中
相容端點，但仍走既有協定 確認 request / response 是否真的相容，並補 profile。 中
新的 provider protocol 新增 adapter、streaming parser、tool normalization、usage 抽取。 高
新的 family + 新的 protocol 除了新 adapter，還要重新思考 fallback、能力宣告與測試策略。 很高

同一個任務，四種協定的基本架構差異

先用一個最小任務當基準：review src/lib.rs，必要時先呼叫 read_file。接著看這個任務在四種協定裡，最外層 body 怎麼長。

協定 主端點 對話承載欄位 工具宣告欄位 第二輪如何延續
OpenAI Responses /responses input tools with type=function 靠 previous_response_id 與 function_call_output
OpenAI Chat Completions /chat/completions messages tools[].function 重送完整 messages 歷史
Gemini GenerateContent :generateContent / :streamGenerateContent contents functionDeclarations 把 tool result 轉成 functionResponse
Anthropic Messages /v1/messages messages tools[].input_schema 以 tool_use / tool_result block 延續

案例一：第一輪請求，四種 body 已經不一樣

在第一輪，大家看起來都像是在送「使用者輸入 + 工具定義」，但實際 JSON 形狀差很多。這就是 請求格式調整 成本的來源。

這裡最值得記住的事

光是第一輪的工具宣告，Aixlarity 就要分別處理 parameters、function.parameters、functionDeclarations、input_schema 四種形狀。這還只是送 request，還沒開始處理 streaming 與 tool result。

案例二：模型先呼叫 read_file 後，第二輪怎麼接回去

真正能看出協定差異的，通常不是第一輪，而是工具跑完之後。這時候 執行時 要把 tool result 回送給模型，讓它在同一輪工作中繼續推理。四種協定在這裡就更不一樣了。

協定 Aixlarity 目前怎麼回送 tool result 這代表什麼
OpenAI Responses 如果已有 previous_response_id，第二輪只送 trailing tool outputs。 這是最明顯的 stateful protocol 形狀，不必每次重灌完整歷史。
OpenAI Chat 重送完整 messages，包含 assistant tool call 與 tool role message。 延續性主要靠完整訊息歷史，而不是 server 端 response id。
Gemini 把 tool result 轉成 user role 的 functionResponse part。 模型會在 contents 中收到工具回應，而不是獨立 tool role。
Anthropic assistant 會先有 tool_use block，下一則 user message 再帶 tool_result。 工具呼叫與結果都被當成 content blocks，而不是外掛欄位。
教學性示意：OpenAI Responses 第二輪為什麼特別薄 
{
  "model": "gpt-5.1",
  "previous_response_id": "resp_123",
  "input": [
    {
      "type": "function_call_output",
      "call_id": "call_1",
      "output": "{\"content\":\"fn main() {}\"}"
    }
  ]
}

這是 Aixlarity 在 OpenAI Responses 路徑裡最值得觀察的一個決策。如果已拿到 previous_response_id，openai_response_input() 會優先只送 trailing tool outputs，而不是把整段歷史訊息重新展開。這意味著 Responses path 在第二輪之後，理論上可以比 Chat path 更簡化，也更依賴 provider 自己保存前態。

教學性示意：OpenAI Chat 第二輪仍保留完整對話歷史 
[
  { "role": "user", "content": "review src/lib.rs" },
  {
    "role": "assistant",
    "content": "",
    "tool_calls": [{
      "id": "call_1",

  }
]

{

  "parts": [{
    "functionResponse": {
      "name": "call_1",
      "response": { "content": "fn main() {}" }

  }]
}

  "role": "assistant",
  "content": [{

案例三：Streaming 差異，比 request body 更容易踩雷

如果說 request body 差異還能靠對照文件慢慢補，那 streaming parser 的差異通常更容易在真實工程裡出事。原因是工具參數常常會以 delta 形式碎片化出現，而 usage 也不見得在同一個事件裡回來。

協定 Aixlarity 目前在等什麼事件 教學上的重點
OpenAI Responses response.output_text.delta、response.function_call_arguments.delta、response.completed 文字與工具參數是分開流的，usage 在完成事件才比較完整。
OpenAI Chat delta content、delta tool_calls、額外 usage chunk 如果沒開 stream_options.include_usage，你可能拿不到流式 usage。
Gemini candidate parts 裡的 text 與 functionCall，以及 usageMetadata 工具呼叫不是獨立事件型別，而是長在 parts 裡。
Anthropic content_block_start、content_block_delta、content_block_stop、message_delta tool JSON 常被分片送來，要自己把 partial_json 拼回去。
整合成本最容易被低估的地方

很多人會以為接好非串流路徑就差不多了，但只要你真的把 streaming 打開，工具參數增量、usage 回補、結束事件、錯誤事件、以及跨 turn continuation 全都會冒出來。這也是為什麼 yage.ai 那類分析會說，模型整合成本遠不只是 API wrapper。

Fallback 目前實際上怎麼工作

Aixlarity 的 fallback 有兩層值得分開看。第一層是「哪些錯誤算可重試」；第二層是「哪些 provider 會被放進候選清單」。這兩層都已經有主要流程實作，但都還有值得保守描述的地方。

層次 目前行為 實況檢查
可重試錯誤判定 靠錯誤字串比對 429、500、502、503、timeout、connection refused 等關鍵字。 ，但屬於偏務實的簡化策略。
fallback 候選清單 目前會收集所有「不是主 provider 且能成功讀到 API key」的 profile。 ，因為註解說 same-family，但實作尚未真的做 family 過濾。
fallback 事件追蹤 會寫入 ProviderFallback 事件，記下 from / to provider 與原因。
教學性示意：fallback 判定大概在做什麼 
主 provider 先跑一次

如果錯誤訊息包含：
- 429
- rate limit
- 500 / 502 / 503
- timeout / timed out
- connection refused

=> 才會進 fallback 流程

接著：
遍歷其他 provider profiles
只要不是主 provider，且能讀到對應 API key
就放進候選清單

這裡最值得帶走的，不是「fallback 很厲害」，而是 fallback policy 本身也是 外殼 設計。如果未來要做得更保守，可能會改成同 family 優先、同 protocol 優先，或至少明確排除風格差太大的跨 family 退路。現在的實作則比較偏向「只要可用，就先保住任務不中斷」。

同一個 fallback 事件，對工程師意味著什麼

F1

功能面

它提高了成功率，避免整輪任務因單一 provider 抽風而立刻失敗。

F2

成本面

不同 provider 可能有不同計價、不同 token 行為與不同回覆風格，因此 fallback 不是免費午餐。

F3

可解釋性

如果沒有 ProviderFallback 事件，團隊很難知道「為什麼今天同一任務忽然變得不一樣」。

F4

產品取捨

你要的是「盡量不中斷」還是「盡量保持模型風格一致」？這會直接影響 fallback policy。

如果今天要接一個新模型，你該先判斷它屬於哪一類工作

情境 你該先檢查什麼 通常要改哪些地方
只是換新的 OpenAI 相容端點 它真的支援 Responses 還是只支援 Chat？tool payload 長得是否一致？ 多半是新增 profile，必要時修補 headers 或 base URL。
接一個新的 Gemini 相容服務 是否仍接受 generateContent / streamGenerateContent 這組形狀？ profile 為主，少量 adapter 相容修補。
接一個完全新的協定 工具宣告、工具結果、streaming 事件、usage 欄位怎麼長？ 要新增 adapter、parser、測試與可能的新 protocol enum。
接一個看似相容、但串流事件不同的供應商 非串流與串流是否都真的相容？ 往往不是只改 profile，而是要動 parser 與錯誤處理。
給貢獻者的一個判讀捷徑

如果你只需要新增一個 providers.conf profile，這通常不是「新模型連接」的 hardest part。真正昂貴的，通常是第二輪工具結果如何回送、流式工具參數怎麼拼回來、以及失敗時該不該 fallback 到另一條完全不同的協定。

實況檢查：這一章最想幫你分清楚的三件事

Profile 不是 protocol

加一個 profile 有時只是在既有 adapter 上多一個設定入口；新增 protocol 則是另一個層級的工程。

Fallback 已經能工作，但 policy 還不算精雕細琢

目前的 fallback 會保住可用性，但 family-aware、cost-aware、style-aware 的排序策略還沒有真正實際做出來。

Streaming parser 是整合成本的大頭

只接好非串流 API 還不夠，真正要拿來做 AI 助手，串流工具參數與 usage 解析往往才是最容易磨人的地方。

把這一章濃縮成一句話

多 provider 的難點不是選單，而是變形與收斂

同一個任務進入 OpenAI Responses、OpenAI Chat、Gemini、Anthropic 之後，會變成不同 request body、不同工具回送形狀、不同 streaming 事件與不同 fallback 風險。Aixlarity 的 provider layer 正是在收斂這些差異，而不是只替 CLI 多加幾個 `--provider` 選項。
