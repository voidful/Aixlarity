Sources

Instructions、Commands、Skills 與 提示詞來源 Layer

這一章專門回答一個常被忽略的問題：agent 最終拿到的 prompt，究竟是從哪些來源一層一層組起來的？

先講結論

對 Aixlarity 來說，prompt 不是只有「使用者輸入」而已。至少還有 repo instructions、custom commands、skills、檔案注入、待批准 shell 區塊與 工作接續。把這些來源講清楚，才能真正理解 AI 助手 為什麼不是單純的聊天模型外掛。

為什麼這一層要單獨看

很多文件在談 prompt 時，只會展示最後送出的文字長什麼樣子。但對工程實作來說，更重要的是「這些文字從哪裡來」。因為只要來源層不清楚，你就很難回答信任規則在哪裡、workflow 是怎麼被固定下來的、哪一層是顯式選擇、哪一層又屬於自動注入。

來源層 它提供什麼 是否受 trust 影響
Instructions repo 級工作規範與語氣 是
Commands 任務模板與 workflow 腳本 是
Skills 可選用的專題能力與長段指引 是
Attachments / File Injections 實際檔案內容、目錄 listing、二進位參照 受工作區解析規則影響
Shell Approval Blocks 把命令意圖轉成待批准區段 不直接執行，進 prompt 顯示
Session Continuation 既有工作摘要 由 session reuse mode 決定

Instructions：repo 規範怎麼進 prompt

InstructionBundle::load() 目前會在可信工作區中讀取四種固定檔名：AGENTS.md、GEMINI.md、CLAUDE.md、AIXLARITY.md。這裡有兩個很重要的訊號。第一，instructions 是顯式檔名約定，不是任意掃描所有 Markdown。第二，它明確受 信任檢查 影響，不可信工作區會直接返回空 bundle。

範例：repo instructions 會如何出現在 最終提示詞 
# Repository Instructions

## AGENTS.md
- 修改前先理解現有結構
- 變更後要跑對應測試

## AIXLARITY.md
- 使用繁體中文撰寫文件
- 不要把基本架構寫成完整功能

這也是為什麼 trust 不只是 UX 標籤。只要 instructions 會進 最終提示詞，它就已經屬於 agent 行為的重要來源，不能不經檢查就相信任何工作區檔案。

Commands：workflow 不是口頭習慣，而是可檔案化模板

CommandCatalog::load() 會讀 global commands dir 與 workspace commands dir 的 TOML 檔，並把路徑轉成 command name。像 review/security.toml 這類路徑會被命名成 review:security。這讓命令不是平面的字串清單，而是可以有層次的 workflow catalog。

檔案位置 最後命令名 說明
commands/review.toml review 最單純的單檔命令。
commands/review/security.toml review:security 可用目錄層次建立命令分類。
.aixlarity/commands/refactor/rust.toml refactor:rust 工作區內的專案特化命令。
範例：一個 custom command TOML 
description = "Review a Rust file with project rules"
prompt = """
Review the target file carefully.
Use @{README.md} as project context.
Task: {{args}}
Run !{cargo test -p aixlarity-core}
"""

命令模板最有價值的地方，不只是省打字，而是把「一類任務應該怎麼展開」變成可以版本化、可以審查、可以重用的文字內容。

{{args}} 與 Raw command 的行為差異

render_command_prompt() 會先做 {{args}} 置換。如果模板裡沒有這個 placeholder，且使用者確實帶了參數，Aixlarity 會把整條原始命令追加為 Raw command。這個細節很實用，因為它避免了「模板忘了留 args 位置，結果使用者參數悄悄消失」的問題。

範例：沒有 {{args}} 時會發生什麼 
模板：
"Review the current file carefully."

使用者輸入：
/review src/app.rs

最終會變成：
Review the current file carefully.

Raw command: /review src/app.rs

Skills：顯式載入的專題能力包

SkillCatalog::load() 會掃 global 與 workspace skills 目錄下的 Markdown 檔。如果檔名是 SKILL.md，名稱取自父目錄；否則取自檔案路徑。summary 則來自第一個非空行。這讓 skill 更像具名能力包，而不是散落 prompt 片段。

檔案位置 最後 skill 名稱 summary 來源
skills/code-review/SKILL.md code-review SKILL.md 第一個非空行
skills/rust/testing.md rust:testing testing.md 第一個非空行

Aixlarity 目前對 skills 的取態很保守：它們是顯式選用，不是靠看不透的 判斷規則s 自動召回。這個設計選擇非常重要，因為它讓 prompt 來源可追蹤。讀者與使用者都能清楚知道：這一輪為什麼帶了這個 skill，而不是系統偷偷決定。

Attachments 與 file injection：證據怎麼進 prompt

expand_file_injections() 會處理 @{...} 語法。如果指向文字檔，會把內容做成帶行號預覽；如果指向目錄，會展成 listing；如果是圖片或 PDF 這類已知二進位類型，則保留成 binary reference metadata。這一層的關鍵，不只是把檔案貼進 prompt，而是把「證據如何被壓縮與呈現」做成可控規則。

A1

文字檔會帶行號

這讓後續 review 或 patch 討論能更精準對齊到檔案位置。

A2

有大小限制

文字注入受 400 行與 32KB 上限控制，避免 prompt 無限制膨脹。

A3

目錄不是整棵樹都展開

listing 會限制數量，且略過 .git 這類不適合直接放進 prompt 的內容。

A4

二進位附件目前是保守連接

會形成 attachment metadata，但最終是否被各 provider 完整利用，仍要保守標成 Implemented。

範例：檔案注入後大概會長什麼樣子 
原始模板：
Use @{src/app.rs} and @{docs/architecture.md} as context.

展開後：
Use
<injected-file path="src/app.rs">
1  fn main() {
2      ...
</injected-file>

and
<injected-file path="docs/architecture.md">
1  # Architecture
2  ...
</injected-file>
as context.

!{...} 不是立即執行，而是 shell approval 區塊

sanitize_shell_blocks() 會把一般文字段中的 !{...} 轉成 <shell-approval required="true">...</shell-approval>，並把命令收集到 pending list。這一層非常關鍵，因為它把「命令建議」和「命令執行」清楚切開。模板可以表達 workflow 中合理的 shell 步驟，但執行仍要交由後續 gate 決定。

範例：shell block 在 prompt 裡的樣子 
原始模板：
Run !{cargo test -p aixlarity-core}

進 最終提示詞 後：
Run <shell-approval required="true">cargo test -p aixlarity-core</shell-approval>

同時 pending_shell_commands 會記下：
- cargo test -p aixlarity-core

一個容易被忽略的 實況檢查：trust 對 commands / skills 的影響比想像大

在 Aixlarity 目前的實作裡，CommandCatalog::load() 和 SkillCatalog::load() 一旦遇到 trust.restricts_project_config()，就會直接回傳空 catalog。這代表在 untrusted workspace 下，不只是 workspace-local 命令與技能受影響，整體 command / skill catalog 都會被縮減。這是一個很重要所以需要寫清楚的行為，因為它直接改變 prompt 來源層。

情境 會發生什麼 對讀者的意義
trusted workspace instructions、commands、skills 都能進 提示詞來源 layer。 agent 能表現出完整 repo workflow 與 skill 組合能力。
untrusted workspace instructions 變空，commands / skills catalog 也會被縮減為空。 agent 仍可工作，但會更接近通用型 coding assistant，而不是專案特化 agent。

把這一層濃縮成一句話

Prompt 來源層不是單純把文字串起來，而是工作流管理

Instructions 把 repo 規範寫進 runtime，commands 把任務模板檔案化，skills 帶入可選用的專題能力，attachments 提供局部證據，shell blocks 把操作意圖延後到批准流程。把這些東西分開，你才真的看懂一個 AI 助手 為什麼比聊天模型難做得多。
