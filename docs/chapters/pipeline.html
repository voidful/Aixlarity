完整流程

從你打一句話，到 AI 真的改了你的檔案，中間經過哪些步驟？這一章帶你走一遍完整流程。

完整的執行流程

這條流程回答四個問題：① 哪些設定在 AI 回答前就決定好了？② 提示詞是怎麼組裝出來的？③ 哪些操作會在執行前被安全檢查擋住？④ 哪些結果會被保存？

Aixlarity 的好處是，這些步驟在原始碼裡都看得到。你不用猜「大概會有信任檢查」，而是能直接找到 Workspace::discover、TrustStore::evaluate、assemble_prompt、run_agent 這些函式。

Step 1

工作區與信任判定

Workspace::discover 找根目錄，TrustStore::evaluate 決定可不可以讀 repo instructions 與 workspace config。沒有這一層，AI 助手可能把不可信的 prompt 當成操作規則。

workspace.rs / trust.rs
Step 2

載入 provider 與 catalog

App::exec 會載入 providers、commands、skills、instructions、plugins，並把這些變成可用的 執行設定。這一步決定 agent 的可用工具與上下文來源。

app.rs / providers.rs / commands.rs / skills.rs
Step 3

組裝最終 prompt

專案規範、命令模板、技能包、附件等資訊，會被組合成一份完整的提示詞送給 AI。這是整個系統最核心的一步——把不同來源的資訊合併成一次請求。

prompt.rs / prompt/expansion.rs
Step 4

呼叫 provider，必要時 fallback

轉接器（adapter）會按照不同 AI 模型的 API 格式發送請求，並解析回覆。如果某個模型暫時故障，系統可以自動切換到備用模型。

agent/adapters/
Step 5

工具執行前的三層 gate

AI 提出的工具呼叫不會直接執行。系統會先檢查三件事：沙盒（在哪裡跑？）、權限（需不需要先問你？）、Hooks（有沒有自訂檢查規則？）。這一層把「AI 想做什麼」和「系統允許什麼」分開。

agent/permissions.rs / hooks.rs / tools/context.rs
Step 6

Session 與事件持久化

執行結果會保存到工作記錄裡：包括提示詞、回覆、工具呼叫次數、token 用量等。這些記錄讓你可以回放、接續、分叉之前的工作。

session/storage.rs / output.rs
01

Workspace 與 Trust

先判定哪個目錄是專案邊界，再決定 repo 指令可不可以影響 agent。

02

Catalog 載入

provider、commands、skills、plugins 與 instructions 在這裡被併成執行前狀態。

03

Prompt 組裝

把任務、附件、命令模板與 repo instructions 寫成可送出的最終 prompt。

04

Provider 執行

adapter 依協定解析 response / SSE，並在需要時切 provider。

05

Permission 與 Tool Gate

模型提出 tool call 後，系統才開始判斷是否允許執行與是否需要人工確認。

06

Session 回寫

把 prompt、結果與事件保存下來，讓未來的回放、fork 與教學都站得住腳。

讀這條 pipeline 時，要特別看哪四件事

A

輸入不是直接送模型

真正的輸入其實包含工作區根目錄、trust 狀態、provider profile、指令檔、命令模板、技能與附件。使用者打出的那一行文字，只是其中一塊。

B

工具不是模型自己執行

模型只會提出 tool call；真正是否執行、怎麼執行、在哪個 sandbox 執行，都由 執行時 決定。

C

安全界線分成多層

Trust 決定哪些設定可以進 prompt；sandbox 決定工具能碰哪些資源；permission 決定某個 call 要不要人工批准。三者不是同義詞。

D

session 是下一輪工作的基礎

Aixlarity 不是把 session 當旁邊的紀錄檔，而是能在 resume / fork 時把摘要重新插回 prompt，讓 workflow 有延續性。

用一個具體任務走一次 pipeline

假設使用者輸入 /review src/app.rs，我們可以把整條路徑拆得很具體：

第一步：命令文字先被翻成 workflow 模板

assemble_prompt 會先試著解析 slash command。如果符合 /review，Aixlarity 不是把這行文字原封不動丟給模型，而是從命令目錄找對應模板，再把 {{args}} 換成 src/app.rs。

第二步：repo instructions 決定回答風格與工作方式

如果工作區可信，像 AGENTS.md 這類檔案就會在 提示詞前段被載入。這一層的作用是把團隊規範、提交方式、測試要求、語言風格變成 AI 可遵守的行為規範。

第三步：檔案與 shell 區塊被轉成受控上下文

如果命令模板內含 @{...} 檔案注入，Aixlarity 會展開成有行號的文字片段或目錄 listing；如果包含 !{...} shell 區塊，則不會直接執行，而是被轉寫成 待審核指令 與 approval 區段。

第四步：模型提出 tool call，runtime 才開始真正做事

當 provider response 解析出 tool calls 後，run_agent 會逐一檢查 permission、執行 hooks，再交給具體工具。這裡可以清楚看到「模型請求」與「系統操作影響」之間有一層可以看到的機制。

第五步：工具結果回傳為下一輪決策基礎

每個工具結果都會被轉成 工具回覆訊息 推回對話狀態，讓模型在下一輪根據真實檔案內容、指令輸出與錯誤訊息再做判斷。這才是 工具迴圈，而不是只做一次 工具呼叫展示。

第六步：整輪執行被寫入 checkpoint、events 與 session turn

如果使用者開啟 checkpoint 或 工作記錄保存，Aixlarity 會把 prompt、tool 次數、token 用量、events 與最終回答一併保存。這使得失敗執行也能被 replay，而不只是留下模糊印象。

範例：/review src/app.rs 可能被組成的 prompt 形狀 
# Runtime
Provider: Gemini 2.5 Pro
Sandbox: workspace-write
Trust: trusted

# Repository Instructions
## AGENTS.md
- 修改前先理解現有結構
- 變更後要跑對應測試

# Task

# 待審核指令
- cargo test -p aixlarity-core

三個很實務的情境例子

T1

不可信倉庫想塞規則給 agent

如果陌生 repo 內含一份要求「先執行安裝腳本」的 AGENTS.md，信任檢查 會先決定這份檔案能不能進 prompt。沒有這層，風險不是模型回答怪，而是整個 workflow 被外部專案重寫。

T2

模型要求套 patch

在 suggest 或 auto-edit 模式下，apply_patch 是否需要人工確認會不同。這讓「我願意讓 agent 幫忙到哪裡」能被明確表達，而不是只剩全開或全關。

T3

主 provider 臨時失敗

如果某個 provider 在可重試錯誤上失敗，fallback provider 清單可以接手。這不是多模型展示，而是讓 執行時 不至於因單點供應商問題整輪中斷。

Trust / Sandbox / Permission 模擬器

這個模擬器想傳達的是：「安全」在 AI 助手 裡不是單一開關，而是三種不同問題的疊加答案。Trust 問的是「這個工作區的規則能不能信」；sandbox 問的是「工具可不可以碰這些資源」；permission 問的是「這次呼叫要不要人工批准」。

    Gate 它攔的是什麼 在 Aixlarity 的原始碼依據 如果沒有這層會怎樣
    Trust 工作區 instructions、skills、workspace providers、workspace commands 是否能影響 agent。 trust.rs、instructions.rs、各 catalog loader 未知 repo 只要放一份指令檔，就可能重寫 AI 的行為的優先順序。
    Sandbox 工具能否寫檔、是否能進容器、能否在工作區外操作。 config.rs、tools/、tools/container.rs 模型提出的操作影響會直接落到主機工作區，失去隔離與可預測性。
    Permission 這次 tool call 需不需要人類點頭。 agent/permissions.rs 使用者無法在高風險操作前介入，agent 也缺少在工作過程中逐步建立信任的方式。

    提示詞是怎麼組裝出來的

    「組提示詞」聽起來好像只是把幾段文字串在一起，但實際上做了三件事：①　把不同來源的資訊（專案規範、任務模板、技能包、附件、歷史記錄）整合起來；②　標記哪些操作需要審核；③　把所有東西轉成 AI 模型 API 能吃的格式。

    來源：instructions.rs

    當工作區被視為可信，Aixlarity 會讀 AGENTS.md、GEMINI.md、CLAUDE.md、AIXLARITY.md。如果沒有 信任檢查，惡意專案就能直接把模型操作規則塞進執行流程。

    來源：commands.rs、prompt/command_invocation.rs

    使用者輸入如果以 / 開頭，會被視為 自訂命令呼叫，並把 {{args}} 替換進模板。這讓 workflow 可以被文件化，不必每次都重打長 prompt。

    來源：skills.rs

    技能目前是顯式選用，而不是自動推斷。這代表 Aixlarity 比比較保守，避免把 技能注入 變成難追蹤的看不見的提示詞演算法。

    來源：prompt/expansion.rs

    @{...} 會把文字檔做行號注入、把目錄展成 listing，二進位檔則記成 attachment metadata。文字注入很完整；二進位附件是否真正被各 provider adapter 使用，則要保守標為 Implemented。

    來源：session/render.rs、app.rs

    resume / fork 時，系統會在 最終提示詞 前插入 工作接續摘要。這使 Aixlarity 的 session 不是單純歷史檔案，而是會回傳到後續決策的 執行時的上下文。

    來源：prompt.rs

    Repo Instructions 的作用：把團隊規範變成 執行時 契約

    這些檔案不是參考資料，而是會進入 最終提示詞 的重要上下文。這意味著 repo instructions 的載入條件與信任檢查，實際上就是安全設計的一部分。

    Commands 的作用：把 workflow 檔案化

    一個 slash command 的價值，不只是少打一點字，而是把常見任務的輸入結構、附件需求與 shell 建議固定下來。這會直接降低團隊工作流的不一致性。

    Skills 的作用：帶入可選用的專題能力

    Aixlarity 目前採顯式選用，而不是自動幫你猜。這個設計的取捨是可追蹤性優先，讓你知道是哪一個 skill 進了 prompt，而不是把 技能注入 變成看不透的推薦機制。

    Attachments 與 Session Context 的作用：讓一次任務不再只是當下輸入

    附件解決的是「模型看不到檔案」；工作接續 解決的是「模型忘了上次做到哪」。兩者都在補足單輪對話天生的上下文不連續。

    你也可以把 提示詞組裝 想成一個「資訊整合站」。它不是任意亂加東西，而是在做資訊分工。repo instructions 提供團隊規範，commands 提供任務模板，skills 提供專題能力，attachments 提供局部證據，工作接續 提供歷史摘要。當這幾種來源混在一起時，才值得特別要求可追蹤與可以觀察。

    Tool call 從 request 到 操作影響，中間經過了什麼

    1

    Provider 先把回應解析成 tool call

    不同 provider 的 function calling / tool use 協定不同，因此 Aixlarity 需要 adapter 去把它們整理成統一的 ToolCall 形狀。

    2

    Permission 先決定需不需要問人

    suggest、auto-edit、full-auto 對不同工具有不同要求。這層把「授權範圍」和「模型能力」切開。

    3

    Hooks 能在工具前後插入額外政策

    PreToolUse / PostToolUse 讓團隊可以在不改動每個工具的情況下，加上額外檢查、紀錄或攔截規則。

    4

    結果被寫回對話與事件流

    工具結果不只回給模型，也被寫進 工具記錄 與 events。這讓後續 replay 與教學回放有具體材料，而不是只剩最終回答。

    如果讀者只看到 其實還應該追問 原因
    「模型呼叫了 shell」 在哪個 sandbox？有沒有 permission prompt？有沒有 hooks？ 這些條件會直接改變同一個 tool call 的安全意義。
    「provider 回了一個 function call」 不同 provider 的事件格式如何被整理成統一形狀？ 如果沒有這層，runtime 很快就會被供應商綁死。
    「結果保存到 session」 只是存檔，還是會回到下一輪 prompt？ 這區分 logging 與真正的 continuation。

    為什麼切換 AI 模型不是小事

    你可能以為「換個 API key 就好」，但實際上不同模型的工具呼叫格式、串流回覆方式、錯誤處理方式都完全不同。Aixlarity 在 run_agent 裡統一處理這些差異，讓你可以一鍵切換 GPT、Gemini、Claude，而不用改任何其他程式碼。

    連接問題 如果只把它當 API wrapper 會怎樣 Aixlarity 現在怎麼處理
    工具呼叫格式不同 每換一家 provider，就得重寫 tool parsing 與工具回傳格式。 由 adapter 層整理成統一 tool call 形狀，再交給 runtime。
    streaming / response event 差異 UI 與 執行時 會被單一 provider 的事件格式綁死。 在 provider 呼叫與 agent event 之間做了重新包裝，讓 session replay 保有一致事件格式。
    錯誤與 fallback 主 provider 一出錯就整輪失敗，或只能在上層寫大量特判。 Aixlarity 會建立 fallback provider 清單，遇到可重試失敗時交由 adapter 層處理。
    上下文爆量 最終只能靠使用者自己縮 prompt，runtime 沒有自我保護。 estimate_total_tokens 與 compact_messages 會在迴圈中主動處理 上下文預算。
    模型整合比想像中複雜

    把 Provider 轉接、回覆解析、備援切換和提示詞組裝放在同一章，是因為它們都是「讓 AI 助手穩定運作」的必要工程。不是接上 API 就結束了——格式轉換、串流處理、錯誤恢復這些看不見的工作，才是真正花時間的地方。
