Personas

Persona 系統與多智能體協作

如何用 Persona 檔案定義角色、限制工具、讓多個 sub-agent 以不同身份並行協作。

先講結論

Aixlarity 的 persona 系統在 instructions、commands、skills 之上，增加了第四層：角色身份。每個 persona 有自己的系統提示詞、工具限制清單和構成規則。coordinator 可以把不同 persona 分配給不同 sub-agent，讓它們在物理隔離下並行工作。這不是靠 prompt 叮嚀「你是 reviewer，不要寫檔案」，而是引擎級工具過濾——tool list 裡根本沒有 write_file。

為什麼需要 Persona

在沒有 persona 的架構裡，所有 sub-agent 都是同質的。coordinator 可以拆任務、排 DAG、平行執行，但每個子代理都看到完整的工具清單和通用的系統提示詞。這會造成兩個問題：

P1

角色混淆

負責 code review 的 sub-agent 可能忍不住幫你「順手修一下」，反而破壞了 review 的客觀性。

P2

權限失控

安全審計員不該有寫入能力，但如果工具清單沒過濾，模型隨時可能嘗試 write_file 或 apply_patch。

P3

上下文浪費

把所有指引（review 框架、測試策略、安全 checklist）全塞進每個 sub-agent 的 prompt，訊噪比極低。

P4

可追蹤性模糊

replay 時無法區分「這段 review 是誰做的」，因為每個 sub-agent 長得一模一樣。

Persona 的設計靈感來自 Addy Osmani 的 agent-skills 專案。核心原則是：persona 定義角色身份與能力邊界，但 persona 之間絕不直接互相呼叫。所有協作都由 coordinator 或 slash command 編排。

一個 Persona 檔案的結構

Persona 檔案放在 .aixlarity/personas/ 目錄，使用 Markdown + YAML frontmatter 格式，與 skill 檔案的結構一致：

.aixlarity/personas/CodeReviewer.md 
---
name: code-reviewer
description: Senior Staff Engineer conducting five-axis code review.
version: 1.0.0
allowed_tools: [read_file, search_files, list_dir, shell]
composition:
  invoke_via: ["/review", "/ship"]
  never_invoke: [Developer, Architect]
---

# Senior Code Reviewer

You are a Senior Staff Engineer. Your sole function is code review.

## Review Framework
Evaluate every change across five dimensions:
1. **Correctness** — Will this code do what it's supposed to?
2. **Readability** — Can another engineer understand this in 30 seconds?
3. **Architecture** — Does this respect module boundaries and dependency direction?
4. **Security** — Are there injection vectors, secrets, or trust violations?
5. **Performance** — Any O(n²) loops, unnecessary allocations, or blocking calls?

## Rules
- Categorize findings as Critical / Important / Suggestion
- Never write or modify code — review only
- Always include at least one positive observation
- If unsure, flag as "Needs Discussion" rather than guessing

Frontmatter 各欄位的意義

欄位 類型 說明
name 字串 機器可讀名稱（小寫連字號），用於 coordinator 的 persona 欄位匹配。
allowed_tools 字串陣列 引擎級工具限制。空陣列 [] 代表「全部工具可用」。若有值，sub-agent 的 tool list 在 run_agent() 入口就會被物理過濾。
composition 物件 標註這個 persona 會被哪些 slash commands 使用 (invoke_via)，以及它永遠不該直接呼叫哪些 persona (never_invoke)。
description 字串 人類可讀的一句話描述，在 IDE 選單中顯示。

引擎級工具過濾是怎麼運作的

工具限制不是靠 prompt 說「你不可以用 write_file」。它是在 run_agent() 的最前面，直接從 tool list 裡移除不在 allowed_tools 裡的工具。模型甚至不知道那些工具存在。

agent.rs — 工具過濾的實際邏輯 
// run_agent() 入口，工具合併之後立刻過濾
let mut tools = all_tools(plugin_tools, mcp_tools);

if let Some(persona_name) = &options.persona {
    let allowed = get_persona_allowed_tools(
        &options.workspace_root, persona_name
    );
    if let Some(allowed_names) = allowed {
        tools.retain(|tool|
            allowed_names.iter().any(|a| a == tool.name())
        );
    }
}
// 此時 CodeReviewer 的 tools 只剩 4 個，
// write_file、apply_patch 等根本不在清單裡
這和 prompt-based 限制的差異

如果你只在 prompt 裡寫「你是 reviewer，不要修改檔案」，模型有可能忽略這條指示。但如果 tool list 裡根本沒有 write_file，模型連嘗試的機會都沒有。這就是物理隔離 vs 語義隔離的差別。Aixlarity 兩者都做：prompt 裡有角色定義（語義層），tool list 有過濾（物理層）。

Coordinator 怎麼用 Persona

spawn_agent 的 JSON schema 新增了 persona 欄位。當模型呼叫 coordinator 時，可以為每個子任務指定角色：

spawn_agent 呼叫範例：三個角色並行 
spawn_agent({
  "tasks": [
    {
      "name": "review",
      "task": "Review all changed files",
      "persona": "CodeReviewer"
    },
    {
      "name": "test",
      "task": "Analyze test coverage gaps",
      "persona": "TestEngineer"
    },
    {
      "name": "security",
      "task": "OWASP Top 10 audit",
      "persona": "SecurityAuditor"
    }
  ],
  "strategy": "parallel"
})

Coordinator 收到後，會：

載入 Persona 內容

從 .aixlarity/personas/{name}.md 讀取完整文件，作為 sub-agent 的系統提示詞核心。

注入到 sub-agent prompt

persona 文件取代通用的 sub-agent 前言，成為 sub-agent 的身份定義。

設定 sub_options.persona

讓 run_agent() 入口的工具過濾正確生效。

平行協作的隔離與合併：Supervisor + Merge Gate

當 Coordinator 以 strategy: "parallel" 產生多個 sub-agent 時，如果所有 agent 同時對工作區的檔案進行修改，極易產生 Race Conditions 與狀態衝突。因此，Aixlarity 實作了「平行意圖產生，序列化變更 (Parallel intent generation, serialized mutation)」架構：

1

Path-Level ACLs (路徑權限控制)

在任務分配時，可以指定 allowed_paths 與 forbidden_paths。引擎層級的 ToolContext 會直接在 write_file 與 apply_patch 工具中攔截越權寫入，確保每個角色只動自己該動的檔案（例如前端專員不能寫後端 DB Migration）。

2

Git Worktree 獨立沙盒

每個平行執行的 sub-agent 不會直接操作主專案，而是被分配到一個獨立的 git worktree 虛擬目錄。它們可以自由思考、測試與修改，結束後系統會自動提取 git diff 轉為結構化的 PatchProposal，隨後銷毀沙盒。

3

Merge Gate (合併門禁)

當該批次所有 agent 完成後，Coordinator 啟動 Merge Gate 模組。它會收集所有的提案，檢查是否有跨 agent 的檔案衝突，並驗證路徑越權。通過檢驗的 patch 才會依照順序以 git apply 的方式合併回主工作區，留下完整審計軌跡 (Audit Trail)。

這解決了什麼問題？

這讓 Aixlarity 的多智能體架構具備「向外擴展 (Scale out)」的基礎能力。不管平行開 3 個還是 30 個 agent 分析不同模組，最終狀態的修改都會被 Merge Gate 嚴格序列化，保證專案的檔案系統不會因為並發修改而崩潰。

三個 Slash Commands 示範多智能體編排

persona 系統最直觀的應用，是透過 slash commands 自動化多角色協作：

Command 策略 參與的 Personas 用途
/ship parallel（三路平行） CodeReviewer + TestEngineer + SecurityAuditor 提交前品質門禁：匯總三方意見，給出 SHIP / NO-SHIP 判定
/spec sequential（兩步序列） Architect → Developer 先設計再實作：確保有技術設計文件後才動手寫 code
/audit single agent SecurityAuditor 深度安全審計：OWASP Top 10 + 威脅建模 + CVE 掃描
.aixlarity/commands/ship.toml 
description = "Pre-ship quality gate: parallel code review, test analysis, and security audit"
prompt = """Use the spawn_agent tool to run these three tasks IN PARALLEL with strategy "parallel":

1. Task named "review" with persona "CodeReviewer":
   Review all changed files for correctness, readability, architecture,
   security, and performance. Use `git diff` to identify changed files. {{args}}

2. Task named "test" with persona "TestEngineer":
   Analyze test coverage gaps for the changed code.
   Identify missing tests, weak assertions, and untested edge cases. {{args}}

3. Task named "security" with persona "SecurityAuditor":
   Audit changed files for OWASP Top 10 vulnerabilities,
   secrets exposure, injection vectors, and trust model violations. {{args}}

After all three agents complete, aggregate their findings into a unified
Ship/No-Ship verdict:
- If ANY Critical issues → verdict is NO-SHIP
- If only Important/Suggestion → verdict is SHIP WITH NOTES
- If no issues → verdict is SHIP
"""

Persona 的核心安全原則：絕不互相呼叫

每個 persona 的 composition 區段都明確寫了 never_invoke。這不只是建議，而是架構原則：

為什麼 Persona 之間不能直接互相呼叫

如果 CodeReviewer 可以直接呼叫 Developer 來「幫忙修一下」，那 review 的獨立性就被破壞了。更嚴重的是，如果 persona A 呼叫 persona B，B 又呼叫 A，就會形成無限遞迴。所有跨角色協作都必須經過 coordinator 或 slash command，這樣每一層都有明確的入口、上限和退出條件。

Fallback 機制：即使沒有檔案也能運作

如果 .aixlarity/personas/ 目錄不存在或缺少某個 persona 檔案，系統會退回到內建的 fallback 定義。每個已知的 persona 名稱都有：

Fallback 提示詞

instructions.rs 裡有 8 個富文本 fallback，每個約 50 字，涵蓋角色定位與工作準則。

Fallback 工具限制

get_persona_allowed_tools() 為每個已知角色提供硬編碼的 allowed_tools，確保即使檔案不存在，CodeReviewer 仍然無法寫入檔案。

別名匹配

支援多種命名變體：codereviewer、code-reviewer、code_reviewer、reviewer 都會匹配到同一個角色。

如何新增自訂 Persona

只需要在 .aixlarity/personas/ 目錄下新增一個 Markdown 檔案，不需要修改任何 Rust 原始碼：

01

建立檔案

例如 .aixlarity/personas/DatabaseAdmin.md，使用 YAML frontmatter 定義 name 和 allowed_tools。

02

定義工具限制

在 allowed_tools 裡列出這個角色可以使用的工具。留空 [] 代表全部可用。

03

撰寫角色指引

在 Markdown body 裡定義 Review Framework、Output Format、Rules 等。這會成為 sub-agent 的系統提示詞。

04

在 coordinator 或 command 中使用

在 spawn_agent 的 persona 欄位填入檔名（不含 .md），例如 "persona": "DatabaseAdmin"。

把這一章濃縮成一句話

Persona 讓 sub-agent 從「同質的工人」變成「有專長和權限邊界的專家」

透過 .aixlarity/personas/ 目錄，你可以用純 Markdown 檔案定義角色身份、工具限制和協作規則，不需要修改 Rust 原始碼。引擎會在物理層強制執行工具過濾，coordinator 會在委派時注入角色提示詞。這就是從「一個 agent 做所有事」到「多個專業 agent 各司其職」的架構轉折點。
