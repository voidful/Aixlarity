Container

Docker 容器沙盒

GCD 可以用 Docker 容器來隔離 shell 指令的執行。但「用了容器」不等於「完全安全」——這一章講清楚它到底保護了什麼。

先講最重要的一句話

GCD 目前的 sandbox=container，應理解成「shell 工具可改走容器執行」，而不是「整個 agent 已經在完整容器沙盒中運作」。如果文件沒有把這件事講清楚，讀者很容易高估安全範圍。

為什麼 container 值得單獨拆一章

在 AI AI 助手 的世界裡，container 這個詞很容易容易讓人誤以為：好像一旦進了容器，所有操作影響就都被隔離、網路就都被關掉、工作區就不會被污染、外部命令也都變得安全。真實工程通常不是這樣。容器只是一種執行環境策略，真正的效果取決於哪些工具走進容器、工作區怎麼掛載、網路怎麼設定、以及 host 上還有哪些路徑仍直接可寫。

對 GCD 來說，這一點特別關鍵，因為 config.rs 的摘要會把 container 描述成「強隔離，適合不受信任 repo 或高風險自動化」，但實作層目前只有 shell 真的改走 Docker / Podman。這不是矛盾，而是開發進度的問題：設計目標，比現在已接好的路徑更大。因此文件必須用 Implemented 的口氣來說明。

問題 如果不分開講 分開講後的正確理解
容器是否等於完整隔離 讀者會以為所有工具、操作影響與網路都被包住。 目前只有 shell 走容器；其他工具仍多半直接作用於 host 工作區。
容器是否等於不會改到工作區 讀者會以為 shell 在容器裡做的修改不會回到本機 repo。 工作區是以 -v host:/workspace:rw 掛進容器，shell 仍可改到實際 repo。
容器是否等於完全沒網路 讀者會把「容器 network 預設關閉」誤讀成整個 agent 都沒有外連能力。 容器內 shell 預設 --network=none，但 fetch_url 目前不走這條邊界。

先把四種 sandbox 想成四個不同問題

很多人把 sandbox 當成單一強弱軸來理解，但實際上四種策略各自想解的問題不同。只用「誰比較安全」來排序，往往會忽略含義差異。

策略 最核心的作用 典型適用情境
off 幾乎不加環境限制，追求速度與自由度。 完全信任的本機實驗，且你清楚知道 AI 助手可能碰到哪裡。
read-only 把 agent 限制成觀察者，不允許直接修改。 review、架構盤點、陌生 repo 初步閱讀。
workspace-write 允許在工作區內修改，但不鼓勵越界碰 host 其他路徑。 有限度修補、補測試、局部重構。
container 讓 shell 改走容器執行環境，降低 host process 直接暴露。 需要跑 shell，但不想直接在本機 userland 上執行命令。
一個很實用的判斷方式

如果你的主要需求是「先不要讓它動手」，首選通常不是 container，而是 read-only。Container 比較適合的情境是「我願意讓它跑 shell，但想把 shell 放在比較收斂的執行環境」。這兩種需求不一樣。

GCD 的 container 路徑現在到底怎麼走

從原始碼看，shell.rs 會在 ctx.sandbox == Container 時，改呼叫 container::run_in_container()。這表示 container 路徑目前是內建在 shell 工具裡，而不是整個 agent 執行時 被包進容器。換句話說，容器是某個工具的執行後端，不是整體程序宿主。

01

先判斷 sandbox 是否為 container

shell 工具一開始會先拒絕 read-only，接著如果策略是 container，就不走本機 sh -c，而是改走容器命令。

02

偵測可用 runtime

detect_runtime() 會先找 podman，找不到才退到 docker。換句話說，如果兩者都存在，GCD 目前偏好 Podman。

03

組出 docker run / podman run 參數

這一步會決定 image、工作區掛載、工作目錄、網路、資源上限與 timeout。這些不是抽象 policy，而是實際命令列引數。

04

在容器內執行 timeout N sh -c ...

真正的 shell 指令是被放進容器後，再經由 timeout 與 sh -c 執行。這表示 timeout 控制目前也屬於容器命令的一部分。

教學性示意：container run 參數大概長什麼樣子 
podman run --rm --interactive \
  -v /host/project:/workspace:rw \
  -w /workspace \
  --network=none \
  --memory=512m \
  --cpus=1.0 \
  ubuntu:22.04 \
  timeout 60 sh -c "cargo test -p gcd-core"

這段命令其實已經把目前設計說得很清楚了：工作區是掛進去的、權限是可寫的、網路預設關掉、CPU / memory 有固定上限、timeout 預設 60 秒。只要你把這些細節讀出來，就不會再把 container 想像成一個神祕看不透的。

Container 目前解決了什麼

C1

把 shell process 從 host userland 移開

這是目前最直接的收益。對未知 repo 或高風險命令來說，這比直接在本機 sh -c 執行更保守。

C2

提供預設無網路的 shell 環境

如果沒有設定 GCD_CONTAINER_NETWORK=true，容器會加上 --network=none。對「先不要外連」的 shell 驗證很有幫助。

C3

提供固定資源與 timeout

預設有 memory、CPU 與執行時間上限，避免某些 shell 任務失控地吃滿資源。

C4

保留工作區上下文

因為工作區被掛進容器，容器內 shell 仍能看到真實 repo，這讓 cargo test、npm test、rg 之類命令仍有實際工程價值。

Container 目前沒有解決什麼

常見誤解 實際現況 為什麼要小心
「有 container 就不會改到本機工作區」 不正確。工作區是以讀寫掛載進容器。 如果 shell 在容器裡跑 cargo fmt、sed -i 或產生檔案，host repo 仍會被修改。
「所有工具都在容器裡」 不正確。目前只有 shell 這條 built-in path 明確進容器。 write_file、apply_patch 仍直接寫 host 工作區。
「container 代表整個 agent 沒網路」 不正確。容器內 shell 預設無網路，但 fetch_url 目前不受這條路徑限制。 如果把它寫成全域網路隔離，文件就會高估實作。
「container 等於 worktree isolation」 不正確。現在是同一份工作區掛載進容器，不是複製到獨立工作樹。 這兩種設計影響 session、git、路徑與回寫含義，不能混著講。
範例：同一個 shell 在 container 模式下仍會改到 host repo 
gcd exec --sandbox container --permission full-auto \
  "執行 cargo fmt 並回報變更"

會發生什麼：
1. shell 改走容器
2. 容器看到的是 /workspace
3. /workspace 其實是 host repo 的可寫掛載
4. cargo fmt 造成的檔案變動會直接出現在本機工作區

所以：
container 不等於「不會改檔」
它只是「shell 在較收斂的執行環境中改檔」

真正值得注意的 host / container 分工

如果把目前 GCD 的工具按「在哪裡執行」來分類，你會更容易理解 container 隔離層的設計。

工具 / 行為 目前主要在哪裡發生 判讀
shell in container mode 容器內 process，但工作區為 host 掛載 的部分路徑。
write_file 直接寫 host 工作區 即使 sandbox 是 container，仍未改走容器檔案系統。
apply_patch 直接寫 host 工作區 和 write_file 一樣屬於 host-side 修改。
read_file / list_dir / search_files host 工作區路徑解析 重點在工作區邊界控制，而不是容器隔離。
fetch_url host 網路路徑 目前沒有被 container network policy 一起包住。

環境變數讓 container 更可調，但不會自動補齊含義缺口

目前容器路徑可透過幾個環境變數調整，這對本地使用很方便，但它們調的是執行參數，不是完成度本身。

環境變數 影響什麼 你應該怎麼理解
GCD_SANDBOX=container 把 container 設成預設 sandbox。 這只是預設值，不代表其他工具自然變成容器化。
GCD_CONTAINER_IMAGE 覆寫預設 image，否則為 ubuntu:22.04。 你可依 repo 工具鏈調整映像檔，但這不改變 host 掛載邏輯。
GCD_CONTAINER_TIMEOUT 覆寫容器內 timeout 秒數，預設 60 秒。 這能調 shell 執行時間，但不是一般 執行時 watchdog。
GCD_CONTAINER_NETWORK 如果設為 1 或 true，容器內 shell 可使用網路。 這只影響容器內 shell，不等於全域網路政策。

什麼情境真的該選 container

01

你要讓 agent 跑 shell，但不想直接碰 host process 空間

像是跑測試、lint、build、檢查套件版本，這些都很適合先丟進 container path。

02

你接受工作區仍可能被修改

因為工作區是可寫掛載，所以 container 更像「隔離命令執行環境」，不是「保證 repo 不變」。如果你不能接受任何修改，還是回到 read-only。

03

你願意把網路政策分開理解

容器內 shell 預設無網路是好事，但它不會自動幫你收斂 fetch_url。如果你的風險程度在意外連，這一點一定要另外處理。

範例：什麼時候該選 read-only，什麼時候該選 container 
情境 A：我要先 review 一個陌生 repo
=> 優先用 read-only
原因：你此刻最需要的是「不要動手」

情境 B：我要在陌生 repo 先跑 cargo test 看看
=> 可考慮 container + suggest
原因：你需要 shell，但想先把 shell 放到較收斂的執行環境

情境 C：我要補測試並讓 agent 自動 patch
=> 多半用 workspace-write + auto-edit
原因：這時候關鍵問題是工作區寫入與批准節奏，不是容器本身

實況檢查：這一層該怎麼標完成度

能力 判讀 理由
shell 改走 Docker / Podman 已有實際 執行時 檢測、命令組裝與執行路徑。
容器 執行時 偵測與基本資源限制 Podman / Docker 偵測、memory、CPU、timeout 都已寫進命令。
整體 sandbox 皆容器化 讀寫工具、網路工具與 提示詞來源 層還沒有一起進到同一個容器邊界。
以工作樹複製或 snapshot 達成真正隔離 目前沒有獨立 worktree / overlay 機制，不能把它講成既有能力。

把這一章濃縮成一句話

Container 現在保護的是 shell 路徑，不是整個世界

GCD 已經有一條可用的 container shell 路徑，這很有價值；但它目前更接近「把 shell 放進較收斂的執行環境」，而不是「所有工具、操作影響與網路都已完全容器化」。把這條邊界講清楚，才是真的對工程師負責。
