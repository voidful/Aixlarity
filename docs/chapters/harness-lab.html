IDE Lab

用 Aixlarity IDE 看懂 Harness

這一章把 IDE 當成教學實驗室：你看到的每個面板、按鈕、卡片和工件，都對應一個 AI agent 背後的工程責任。

Lab 的讀法

不要先把 Aixlarity IDE 當成「又一個 AI 編輯器」。比較好的看法是：它是一台可以拆開看的 AI 工作機器。前端負責讓人下決定，Rust 背景服務負責執行 agent，工件和證據負責讓每次 AI 行動都能被檢查。

第一站：輸入框不是聊天，是任務建立器

你在輸入框送出的文字，進入背景服務前，IDE 會補上目前工作區、角色、模型供應商、沙盒、權限、選取文字、目前檔案等上下文。這就是 harness 的第一個職責：把「一句話」轉成一個真的可以執行的任務。

從一句話到 JSON-RPC 
使用者輸入
 active editor / selection
 provider / model
 persona
 sandbox / permission
 checkpoint / auto-git

 workspace trust
------------------------------------------------
agent_chat JSON-RPC request → Rust daemon → agent loop

第二站：批准卡片把安全變成互動

提示詞不能當安全邊界。當 agent 想跑 shell、寫檔、套 patch，IDE 會用批准卡片把風險攤開給人看。這裡要學的是：權限是引擎層級的門檻，不是「請模型乖一點」的文字。

UI 行為Harness 概念你該觀察什麼
允許 / 拒絕 / 本輪都允許 一次性與本輪工作階段的權限 同一個工具在不同權限等級下是否需要人批准。
Shell 指令卡片 危險命令證據化 命令、cwd、風險理由、輸出是否都可追蹤。
批准紀錄 可回放的決策紀錄 後續能否知道「誰批准了什麼」。

第三站：Mission Control 管長任務

真正的 agent 產品不能只有一個一直往下捲的聊天窗。Mission Control 把任務、批准、工件、審計、工作區索引集中在一起，讓長任務可以暫停、繼續、取消、重試，也能在 IDE 重啟後恢復。

Task

任務不是聊天訊息

任務有狀態、進度、時間線、工件 ID。這是從聊天產品走向 agent 產品的關鍵。

Audit

決策要能追

批准、工件審查、留言討論都應該能被存回可持久化的狀態。

Resume

重啟後要接得上

Mission Control 狀態存在 .aixlarity/state/mission_control.json，不是只活在畫面裡。

第四站：工件審查讓 AI 修改看得懂

AI 每次修改程式碼，都應該變成結構化工件：名稱、類型、狀態、摘要、內容、證據、附件、討論串。這讓「我看過了」變成產品流程，而不是口頭承諾。

工件類型教學重點對應能力
Code Diff 逐段檢查 AI 修改，每一輪編輯都可比較。 Visual Diff Review、Compare Rounds、Review Gate
終端機紀錄 測試和命令不能只靠模型說明，要有 stdout/stderr/exit code。 Terminal Replay Center
瀏覽器錄製 瀏覽器操作要留下 DOM、console、network、截圖、影片。 Integrated Browser Agent
測試報告 驗收結果要能附回任務，不只是終端機裡的一段文字。 Evidence Bundle、Artifact Export

第五站：模型供應商切換要有秩序

多模型不是把 API key 塞進設定檔而已。Aixlarity 把 provider 分成個人 / 工作區範圍，API provider 必須填 model，匯入匯出包不能帶 raw secret。這些都是開源 IDE 需要的產品品質。

Scope 是團隊協作邊界

工作區 provider 可以跟專案一起分享；個人 provider 留在自己的環境。這能避免把私密 API 設定誤送進 repo。

Model 必填是可重現性

同一個 provider family 下的模型差異很大。沒有 model ID，就很難重現 agent 行為。

Bundle 不帶 key 是安全設計

匯入匯出應該分享 provider 設定形狀，不應該分享 secret。這是開源工具特別重要的底線。

第六站：讓 agent 進入日常開發位置

Harness 不該只存在側邊欄。Aixlarity 把 agent 動作放到 diagnostic hover、Problems panel、選取文字、終端機輸出，讓使用者可以把具體上下文送給 agent，而不是手動描述一大段。

Editor-native agent flow 
Diagnostic hover → Explain and fix
 Problems panel → Send all to Agent
 Selection → Explain / Fix / Review
 Terminal output → Send to Agent
------------------------------------------------
把精準上下文交給 harness，而不是讓使用者重打問題。

Lab 作業：用一次 AI 修改學完整個 Harness

01

開一個小任務

在 IDE 裡請 agent 修改一個小檔案，先用 suggest permission，觀察 approval card 何時出現。

02

檢查 diff 工件

進 Evidence，打開 code diff，切換 side-by-side / unified，對每一段修改做批准或留言。

03

要求 agent 跑測試

觀察終端機紀錄是否留下 command、cwd、exit code、stdout/stderr，並把結果附回工件。

04

重啟後回到 Mission Control

確認任務、工件、審查討論串是否仍在。這一步檢查的是可持久化狀態，不是 UI 動畫。

把 Lab 帶回團隊

如果要用 Aixlarity 對外教學或內部 workshop，最有效的講法不是「我們有 AI IDE」，而是「我們把 AI agent 的黑盒拆成可驗證的任務、工具、證據、diff、模型供應商和安全邊界」。
